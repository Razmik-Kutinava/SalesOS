# Участие в разработке SalesOS

Краткие правила для коммитов, веток и качества кода. Если вы работаете один — это чеклист для будущего «мы».

## Ветки

- `main` (или `master`) — стабильная линия; прямые пуши по договорённости команды.
- Фичи: `feature/kratkoe-opisanie` или `feat/...`.
- Исправления: `fix/...`.
- Документация только: `docs/...`.

## Коммиты

- Сообщения на **русском или английском** — единообразно в рамках репозитория (сейчас допустимы оба; предпочтительно осмысленное предложение).
- Формат по желанию: префиксы `feat:`, `fix:`, `docs:`, `chore:`.
- Один логический коммит — одна цель; избегайте «мешанины» рефакторинга и фичи без необходимости.

## Pull request

- Описание: **что** сделано и **зачем**; ссылки на задачу/PRD при наличии.
- Самопроверка: тесты, линтер, быстрый ручной сценарий если затронут UX.

## Ruby / Rails

- Следовать существующему стилю файлов; не рефакторить «заодно» несвязанный код.
- Запуск тестов: `bin/rails test` (и при появлении — `bin/ci` если добавлен).

## Линтер

- RuboCop: `bin/rubocop` или `bundle exec rubocop`.
- Исправления авто там, где уместно: `bin/rubocop -A` (проверить дифф глазами).

## Документация

- Изменения в поведении продукта или API — обновить соответствующие файлы в `docs/` в том же PR, если это разумно по объёму.
- Крупные архитектурные решения — новый файл в `docs/ADRs/`.

## Секреты

- Не коммитить `.env`, ключи API, дампы БД с реальными ПДн.
- Примеры конфигурации — только с плейсхолдерами.

## Troubleshooter

- Повторяющийся инцидент с чётким решением — запись в `docs/troubleshooter/` по шаблону.

## Вопросы

- Продукт и scope: `docs/product/PRD.md`
- Роли и стыковка: `docs/devdep/`

Спасибо за аккуратные изменения.
