# Security, комплаенс, технический писатель

Три зоны в **одном документе**: безопасность разработки и эксплуатации; внешний комплаенс и юридический контур; документация для пользователей и команд.

---

## Часть 1. Security / AppSec-инженер

### 1.1. Миссия

Снизить риск **взлома, утечек данных, злоупотреблений** и регуляторных санкций через проактивные меры, обучение и отработанную реакцию на инциденты.

### 1.2. Обязанности

**Моделирование угроз.** Для ключевых сервисов: кто атакующий, какие активы, какие векторы. Упрощённые STRIDE/LINDDUN или внутренний шаблон.

**Secure SDLC.** Точки встраивания проверок: дизайн, ревью, сканирование в CI, пентест перед крупным релизом.

**Сканирование.** Зависимости (SCA), статический анализ (SAST), образы контейнеров, секреты в git (secret scanning).

**Ревью критичных изменений.** Аутентификация, сессии, платежи, загрузка файлов, экспорт данных, админки.

**Управление уязвимостями.** Очередь findings, severity, SLA на патч, исключения с обоснованием и сроком пересмотра.

**Инциденты безопасности.** Playbook: изоляция, сохранение артефактов, расследование, уведомления (в т.ч. регулятор при необходимости — с юристом).

**Обучение.** Короткие сессии для разработки: OWASP Top 10, безопасная работа с секретами, фишинг.

### 1.3. Артефакты

Политики безопасности, отчёты сканеров, тикеты на remediation, постмортемы инцидентов, матрица доступов (высокоуровневая).

### 1.4. Взаимодействие

Tech Lead (архитектурные компромиссы), DevOps (сеть, секреты, логи), Backend (код и данные), DBA (доступы к БД), юрист (ПДн, уведомления).

### 1.5. Границы

Не **пишет** все фиксы сам — выдаёт требования и проверяет закрытие. Не заменяет **юриста** в юридических текстах. Не является **единственным** владельцем культуры безопасности — это обязанность всей компании.

### 1.6. Навыки

Угрозы, сети, основы кода на уровне чтения, стандарты (ISO 27001 на уровне понимания, SOC2 и т.д. по контексту компании), коммуникация без запугивания.

### 1.7. Антипаттерны

Security как **блокер** без альтернатив; вечный «no» без risk acceptance на уровне руководства; игнорирование usability security (люди обходят правила).

## Часть 2. Комплаенс и юрист (вне DevDep, связь обязательна)

### 2.1. Миссия юридической функции

Соответствие **закону, договорам и внутренним политикам**: ПДн, отраслевые регламенты (финтех, медицина и т.д.), лицензии ПО и контента.

### 2.2. Что ожидать от взаимодействия

DevDep готовит **техническое описание**: какие данные собираются, где хранятся, кому передаются, сроки хранения, subprocessors.

Юрист/комплаенс возвращает **ограничения и обязательства**: тексты согласий, DPIA, трансгран, требования к логам и удалению.

Решения о **риске** при невозможности идеального соответствия принимает **руководство** с подписью, не инженер в одиночку.

### 2.3. Границы для разработчиков

Не выдумывать формулировки **юридической силы** для публичных политик без review. Не хранить ПДн «на всякий случай» дольше, чем позволяет цель обработки.

### 2.4. Взаимодействие

PO (продуктовые формулировки), Security (технические меры), DBA (retention, шифрование), поддержка (запросы субъектов данных).

## Часть 3. Технический писатель (Technical Writer)

### 3.1. Миссия

Сделать продукт и систему **понятными**: пользователи реже пишут в поддержку, новые разработчики быстрее вступают в строй.

### 3.2. Обязанности

**Пользовательская документация.** Гайды, FAQ, release notes для клиентов.

**Внутренняя документация.** Онбординг разработчика, runbook для саппорта, архитектурные обзоры в доступном языке.

**API-документация.** OpenAPI/Swagger, примеры запросов и ответов, коды ошибок совместно с backend.

**Единый глоссарий.** Термины продукта одинаково в UI, доке и саппорте.

**Редактура.** Ясность, структура, единый стиль; не обязательно глубокое знание кода каждой строки.

### 3.3. Артефакты

Портал документации, страницы в wiki, PDF при необходимости, changelog.

### 3.4. Взаимодействие

PO (что анонсировать), Backend/Frontend (факты), QA (сценарии для help), поддержка (типовые вопросы), юрист (публичные формулировки).

### 3.5. Границы

Не определяет **архитектуру**. Не подменяет **PO** в продуктовых сообщениях без согласования. Не пишет **юридические** тексты без legal.

### 3.6. Навыки

Структурирование информации, упрощение без искажения, базовое понимание технологий, инструменты docs-as-code при принятии в команде.

### 3.7. Антипаттерны

Документация **отстаёт** на два релиза без процесса обновления; копипаст API без проверки на стенде; игнорирование локализации если продукт мультиязычный.

## Часть 4. Стыки трёх зон

| Ситуация | Кто ведёт |
|----------|-----------|
| Утечка данных | Security + юрист + EM |
| Новая политика cookies | Юрист + PO + Tech Writer |
| Публичный API | Backend + Tech Writer + Security (auth) |
| Пентест | Security координирует, DevDep исправляет |

---

## Часть 5. Итог

**Security** защищает систему и данные. **Комплаенс/юрист** задаёт **рамку закона**. **Tech Writer** переводит сложное в **понятное**. Все три уменьшают **риск репутации и штрафов** и стоимость поддержки.

---

## Часть 6. Bug bounty и внешние исследователи

Если программа есть — Security определяет scope, правила выплат, поток triage отчётов. Разработка получает тикеты с приоритетом.

---

## Часть 7. Локализация документации

При мультиязычном продукте Tech Writer или локализационная команда синхронизируют обновления: не оставлять русскую версию на два релиза позади английской без пометки.

---

## Часть 8. Хранение доказательств

Для возможных споров и расследований Security и юрист определяют, **что** и **как долго** хранить (логи доступа, письма согласий) — технически помогает DevOps/DBA.

---

## Часть 9. Обучение при онбординге

Короткий модуль по безопасности и по работе с ПДн для каждого нового сотрудника DevDep — стандарт зрелой организации.
