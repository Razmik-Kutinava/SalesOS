# Frontend-разработчик, UX-дизайнер, UI-дизайнер

Три роли в **одном документе**: реализация интерфейса в коде; исследование и сценарии; визуальная система. Границы между ними критичны, чтобы не дублировать работу и не оставлять дыры.

---

## Часть 1. Frontend-разработчик

### 1.1. Миссия

Сделать продукт **удобным, предсказуемым и доступным** в клиентской среде (браузер, мобильное WebView, десктоп-оболочка — по продукту). Пользователь понимает, что происходит в каждом состоянии: загрузка, успех, ошибка, пусто.

### 1.2. Обязанности

**Вёрстка и компоненты.** Реализация макетов с соблюдением сетки, типографики, отступов из дизайн-системы. Компонентный подход, переиспользование.

**Работа с API.** Запросы, кэширование на клиенте где уместно (React Query, SWR, собственные паттерны — по стеку). Обработка ошибок сети и бизнес-ошибок (4xx/5xx). Синхронизация состояния форм с сервером.

**Доступность (a11y).** Семантические теги, роли, подписи к полям, фокус и клавиатура, контраст. Соответствие WCAG на уровне, договорённом с продуктом.

**Производительность.** Ленивая загрузка модулей и изображений, минимизация лишних ререндеров, профилирование при проблемах. Размер бандла в SPA-стеках.

**Кросс-браузерность и адаптив.** Тестирование целевых браузеров и размеров экранов; полифиллы по политике команды.

**Участие в ревью.** Визуальные регрессии при наличии скриншот-тестов; внимание к UX-деталям в PR коллег.

### 1.3. Артефакты

Код UI, storybook или аналог (если принят), документация нестандартных компонентов, скриншоты для QA при сложных состояниях.

### 1.4. Взаимодействие

Backend (контракт API, ошибки), UX/UI (макеты, уточнения), QA (баги воспроизведения), PO (приоритет UI-задач через бэклог).

### 1.5. Границы

Не является **единственным** местом критичной бизнес-валидации — сервер остаётся источником истины. Не подменяет **UX-исследователя** на глубоких исследованиях без договорённости.

### 1.6. Навыки

HTML/CSS/JS или фреймворк команды, инструменты сборки, git, основы HTTP, основы a11y, работа с дизайн-токенами.

### 1.7. Антипаттерны

Дублирование сложной логики только на клиенте; «пиксель-перфект» в ущерб доступности; игнорирование состояний loading/error; хардкод текстов без i18n где i18n обязателен.

## Часть 2. UX-дизайнер (User Experience)

### 2.1. Миссия

Снизить **трение** в пользовательских сценариях: меньше ошибок, меньше вопросов «что дальше», выше завершение ключевых потоков.

### 2.2. Обязанности

**Исследования.** Интервью, юзабилити-тесты, анализ воронок совместно с аналитиком. Формулировка проблем, а не только экранов.

**Пользовательские потоки.** Схемы шагов, edge cases (нет прав, нет данных, истёк срок). Прототипы низкой и средней детализации.

**Передача в разработку.** Спецификация состояний экрана; согласование с PO по приоритету сценариев.

**Валидация после релиза.** Сбор обратной связи, гипотезы для следующих итераций.

### 2.3. Артефакты

Прототипы (Figma, FigJam, Miro), user flow, резюме исследований, рекомендации по копирайту (в паре с маркетингом при необходимости).

### 2.4. Границы

Не обязан **верстать** продакшен. Финальный **визуальный** стиль может отдавать UI-дизайнеру. Не определяет **технический** способ реализации (какой фреймворк).

### 2.5. Взаимодействие

PO (цели и приоритеты), UI (визуал), Frontend (реализуемость), аналитик (цифры), пользователи или их представители.

---

## Часть 3. UI-дизайнер (User Interface)

### 3.1. Миссия

Визуальная **целостность** и узнаваемость продукта: типографика, цвет, иконки, компоненты, соответствие бренду.

### 3.2. Обязанности

**Дизайн-система.** Токены, компоненты, состояния (hover, disabled, error). Документация для фронтенда.

**Макеты экранов.** Адаптивные брейкпоинты, плотность информации, согласованность отступов.

**Спецификации.** Что передаётся в разработку: отступы, радиусы, тени, анимации (в разумных пределах).

### 3.3. Границы

Не подменяет **исследование мотиваций** и сложных потоков без UX — фокус на визуале и компонентах.

### 3.4. Взаимодействие

UX (логика), Frontend (реализация), бренд-маркетинг (гайдлайны), PO (приоритет экранов).

---

## Часть 4. Стыки трёх ролей

| Вопрос | Кто ведёт |
|--------|-----------|
| «Пользователь не понимает шаг 2» | UX |
| «Кнопка выглядит не как в системе» | UI |
| «Компонент не совпадает с макетом в пикселях из-за браузера» | Frontend + UI |
| «Нужна новая форма входа» | PO приоритизирует; UX проектирует поток; UI оформляет; Frontend строит |

---

## Часть 5. Совместная работа в спринте

На планировании: UX/UI дают **готовность** макетов или явный **долг** с датой. Frontend не берёт задачу «сверстать экран» без **критериев** и макета или осознанного wireframe-договора.

---

## Часть 6. Инструменты (примеры, не норма)

Figma, Sketch, Zeroheight для дизайн-систем; Maze/UserTesting для тестов; Storybook на стороне разработки.

---

## Часть 7. Метрики успеха (осторожно)

- Снижение **отвалов** на ключевых шагах воронки (аналитика).
- Время **задачи пользователя** (где измеримо).
- **Доступность**: доля критичных нарушений a11y в аудите.
- Скорость **handoff** макет → merge без бесконечных итераций «чуть левее».

---

## Часть 8. Антипаттерны межролевые

- Макеты без состояний **ошибки** и **пусто**.
- UI без согласования с **реальными** данными (длинные строки, другой язык).
- Frontend меняет UX **молча** без согласования с UX/PO.
- «Красиво» в ущерб **читаемости** и контрасту.

---

## Часть 9. Итог

**Frontend** воплощает интерфейс в коде. **UX** защищает логику сценария. **UI** защищает визуальную целостность. Вместе они определяют **ощущение** продукта у пользователя не меньше, чем бэкенд-логика.

---

*В малой команде один человек может совмещать UX+UI или UI+Frontend — границы из документа всё равно полезно держать в голове.*

---

## Дополнение: дизайн-токены в коде

Синхронизация токенов (цвет, шрифт, радиус) между Figma и репозиторием через стиль-пайплайн или ручной регламент обновлений — снижает рассинхрон между UI и Frontend.
