# Backend-разработчик и Full-stack-разработчик

Два связанных профиля в **одном документе**: backend — ядро серверной разработки; full-stack — сквозная реализация от UI до БД. Описание обобщённое, без привязки к языку или фреймворку.

---

## Часть A. Backend-разработчик

### A.1. Миссия

Реализовать **надёжную серверную часть** приложения: бизнес-правила, API, интеграции и фоновые процессы так, чтобы данные оставались **консистентными**, а поведение при сбоях внешних систем — **предсказуемым**.

### A.2. Основные обязанности

**API и контракты.** Проектирование и реализация HTTP/JSON, GraphQL, gRPC или иного протокола по стандарту команды. Документирование контрактов (OpenAPI, схемы). Обработка ошибок: коды, тела, идемпотентность для критичных операций.

**Модель данных и персистентность.** Сущности, связи, валидации на уровне приложения. Транзакции и границы консистентности. Миграции схемы; согласование с DBA на тяжёлых изменениях (блокировки, объём данных).

**Фоновые задачи.** Очереди, воркеры, расписания (cron). Повтор при ошибках с backoff; защита от дублирования side-effect (письма, платежи). Мониторинг очередей.

**Интеграции.** Клиенты к внешним API: таймауты, ретраи, circuit breaker при необходимости. Работа с секретами через конфигурацию окружения, а не хардкод.

**Безопасность приложения.** Аутентификация и авторизация; проверка прав на каждом чувствительном эндпоинте. Защита от инъекций (SQL, NoSQL, command). Ограничение массового присваивания полей. Минимизация PII в логах.

**Наблюдаемость.** Структурированные логи, correlation id, метрики (латентность, ошибки), трассировки — в согласовании с DevOps/SRE.

### A.3. Артефакты

Исходный код, unit- и интеграционные тесты, миграции БД, спецификации API, README по локальному запуску бэкенда, примеры запросов для фронта/QA.

### A.4. Взаимодействие

- **Frontend / mobile:** контракт API, версии, обратная совместимость.
- **DBA:** индексы, медленные запросы, блокировки.
- **DevOps:** переменные окружения, healthcheck, docker-образ приложения.
- **QA:** как воспроизвести сценарий, тестовые данные.
- **Security:** уязвимые места в коде, секреты.
- **PO:** уточнение edge cases (без подмены приоритетов).

### A.5. Границы

- Не подменяет **PO** в определении бизнес-ценности.
- **Критичные правила** не полагаются только на клиент: дублирование валидации на сервере.
- Не меняет **продакшен-схему** без процесса review с DBA/Tech Lead при риске даунтайма.

### A.6. Навыки

Язык и фреймворк команды, SQL, основы распределённых систем, HTTP, безопасность веб-приложений, тестирование, git, code review.

### A.7. Антипаттерны

«Сделаю в проде быстрее» без тестов; гигантские контроллеры без сервисного слоя; N+1 запросы; секреты в репозитории; игнорирование таймаутов внешних вызовов.

### A.8. Рост

От **junior** (задачи по шаблону с ревью) к **senior** (самостоятельный дизайн подсистем, менторинг). Специализация: высоконагруженные сервисы, интеграции, платёжные потоки.

## Часть B. Full-stack-разработчик

### B.1. Миссия

Закрывать **вертикальные срезы** продукта: от интерфейса до хранения данных, когда команда мала или нужен быстрый MVP. Сохранять **ясные границы** между слоями для будущего разделения ролей.

### B.2. Обязанности

Всё из раздела **Backend** в объёме, необходимом для фичи, плюс:

**UI.** Реализация экранов по макетам или дизайн-системе; состояния загрузки и ошибок; базовая доступность.

**Согласование слоёв.** Явное описание API между «своим» фронтом и бэком; избегание неявной «магии» только в голове разработчика.

**Производительность клиента.** Разумный размер бандла (если есть SPA), ленивая загрузка, лишние запросы к API.

### B.3. Границы

При росте команды **не монополизирует** оба направления без синхронизации: передаёт глубокую UI-полировку frontend-специалисту или уходит в backend-фокус.

Не обязан быть **визуальным дизайнером** — использует готовую систему компонентов и макеты UX/UI.

### B.4. Взаимодействие

UX/UI (макеты), Tech Lead (границы модулей), QA (E2E), тот же круг, что у backend, для серверной части.

### B.5. Антипаттерны

«Я и так всё знаю» без документации для команды; смешение бизнес-логики в компонентах UI без выноса на сервер где нужно; пренебрежение тестами на одном из слоёв.

## Часть C. Общее для обоих профилей

### C.1. Code review

Конструктивные комментарии; следование team guidelines; вопросы безопасности и производительности применимы к обоим.

### C.2. Git и ветвление

Feature branches, понятные сообщения коммитов, мелкие PR где возможно — снижает риск и упрощает ревью.

### C.3. Документация

Обновление README и внутренних how-to при изменении способа запуска или интеграции.

### C.4. Онбординг

Помощь новым коллегам в области своего стека; терпение к вопросам «где в коде X».

### C.5. Инциденты

Участие в разборе, если инцидент связан с недавними изменениями в коде; честное описание причин.

### C.6. Этика и данные

Не использовать доступ к прод-данным вне служебной необходимости; соблюдать политику компании по ПДн.

---

## Часть D. Метрики качества работы (ориентиры)

- Низкая доля **регрессий** после релизов в зоне ответственности.
- Покрытие тестами **критичных** путей (не гонка за 100% ради цифры).
- Время **ревью** PR в разумных пределах.
- Отсутствие **повторяющихся** инцидентов одного типа без постмортем-действий.

---

## Часть E. Итог

**Backend** — опора **целостности и безопасности** серверной логики. **Full-stack** — ускоритель в малой команде при дисциплине границ. Обе роли требуют **ответственности** за продакшен-поведение системы, а не только «закрыл задачу в Jira».

---

## Часть F. Типичные задачи по неделе (пример)

- Реализация эндпоинта или джобы, ревью чужих PR, исправление бага по тикету QA, участие в стендапе, короткий синк с фронтом по контракту, написание/правка тестов, обновление миграции при смене схемы.

---

## Часть G. Совместная ответственность с DBA

При добавлении тяжёлого индекса или изменении типа столбца на большой таблице backend инициирует **дизайн** миграции, DBA — **оценку** блокировок и окно выката. Итоговое решение фиксируется в задаче или ADR.

---

## Часть H. Версионирование и обратная совместимость

Политика deprecation: срок жизни старого API, заголовки или версия в пути, коммуникация потребителям (фронт, партнёры). Full-stack при сквозной разработке особенно важен **синхронный** план снятия старого контракта.
