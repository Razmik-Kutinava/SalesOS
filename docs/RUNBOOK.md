# Runbook: деплой, дежурство, инциденты

**Версия:** 0.1  
**Статус:** Операционный каркас (заполнить контакты и URL под среду)

---

## 1. Назначение

Пошаговые действия для развёртывания, отката, проверки здоровья и реакции на типовые сбои SalesOS.

---

| Среда   | URL (пример)     | База данных        | Назначение        |
|---------|------------------|--------------------|-------------------|
| local   | localhost:3000   | docker postgres    | разработка        |
| staging | TBD              | staging DB         | приёмка           |
| prod    | TBD              | managed PostgreSQL | пользователи      |

---

## 3. Предусловия продакшена

- Ruby версия из `.ruby-version`
- PostgreSQL доступен из приложения
- Переменные окружения заданы (см. INTEGRATIONS.md / SECURITY)
- Ollama доступен с сервера приложения (если не на той же машине — сеть/firewall)

---

## 4. Деплой (Kamal / Docker)

**Высокоуровнево:**

1. Сборка образа из `Dockerfile`.
2. Push в registry.
3. `kamal deploy` (или аналог) с zero-downtime при нескольких репликах.

**Проверка после деплоя:**

- `GET /up` → 200
- Smoke: логин тестового пользователя (staging)
- Очередь jobs не залита failed

---

## 5. Миграции БД

```bash
bin/rails db:migrate
```

На проде — в шаге deploy hook после нового образа, до переключения трафика или с advisory lock.

**Откат миграции:** только если миграция обратима; иначе forward-fix.

---

## 6. Откат релиза

1. Переключить Kamal на предыдущий **image tag**.
2. При несовместимой миграции — не откатывать БД с потерей данных; выкатить hotfix.

---

## 9. Логи

- `docker logs` или stdout агрегатор (CloudWatch, Loki, etc.)

---

## 10. Бэкапы PostgreSQL

- **Частота:** ежедневно полный + WAL/continuous по возможности.
- **Хранение:** отдельный регион/бакет с версионированием.
- **Тест восстановления:** ежеквартально на staging.

Команда пример (уточнить под хостинг):

```bash
pg_dump "$DATABASE_URL" | gzip > backup-$(date +%F).sql.gz
```

---

## 11. Восстановление из бэкапа

1. Остановить приложение (maintenance page).
2. Восстановить БД на чистый инстанс.
3. Проверить миграции `db:migrate:status`.
4. Запустить приложение, smoke тест.
5. Включить трафик.

---

## 12. Инцидент: приложение 502/503

**Симптомы:** балансировщик не получает healthy от Puma.

**Действия:**

1. Проверить статус контейнеров / systemd.
2. Логи Puma на ошибки boot (credentials, DB).
3. Проверить `DATABASE_URL` и сетевую связность.
4. Откат образа при недавнем деплое.

---

## 13. Инцидент: БД недоступна

**Действия:**

1. Статус managed DB в консоли провайдера.
2. Connection pool exhaustion — увеличить pool или уменьшить воркеры временно.
3. Disk full — расширить storage.

---

## 14. Инцидент: Ollama не отвечает

**Симптомы:** голосовые команды падают, `llm_unavailable`.

**Действия:**

1. `curl -s OLLAMA_HOST/api/tags` с сервера приложения.
2. Проверить GPU/CPU load, память.
3. Перезапуск сервиса Ollama (осторожно с активными запросами).
4. Включить деградационное сообщение пользователям (баннер в админке).

---

## 15. Инцидент: очередь jobs растёт

**Действия:**

1. Открыть Solid Queue / Sidekiq UI (защищённый).
2. Найти класс job с наибольшим backlog.
3. Проверить failed jobs причину (CRM timeout, Ollama).
4. Масштабировать workers или временно отключить источник нагрузки.

---

## 16. Инцидент: Telegram webhook не доставляет

**Действия:**

1. `getWebhookInfo` через Bot API.
2. Проверить TLS сертификат домена.
3. Проверить 200 ответ < таймаута Telegram.
4. Перерегистрировать webhook.

---

## 17. Инцидент: диск заполнен (uploads)

**Действия:**

1. Найти большие файлы в storage.
2. Запустить задачу очистки временных voice файлов.
3. Расширить volume или перенести на S3.

---

## 18. Инцидент: утечка секрета

**Действия:**

1. Ротировать ключ немедленно.
2. Проверить access logs на злоупотребление.
3. Post-mortem в docs/troubleshooter.
