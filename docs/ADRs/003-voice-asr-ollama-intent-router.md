# ADR 003: Конвейер голос → ASR → Ollama → IntentRouter → действие

**Статус:** Принято  
**Дата:** 2026-03-28  
**Контекст:** Phase 1, основной UX «голосом»

## Контекст

Пользователь должен управлять CRM голосом из браузера и Telegram. Нужен предсказуемый, безопасный путь от аудио до изменений в БД без выполнения произвольного кода из ответа модели.

## Решение

1. Клиент отправляет аудио на сервер (или Telegram file → сервер).
2. **ASR** (Whisper или эквивалент) даёт транскрипт; сохраняется в сессии/событии.
3. **Ollama** вызывается с system prompt (роль, язык, ограничения) и пользовательским текстом = транскрипт + сжатый контекст лида.
4. Ответ модели парсится в **структурированный JSON**: `intent`, `slots`, `assistant_message`, `need_approval`.
5. **IntentRouter** проверяет intent по allow-list, валидирует слоты, применяет политики RBAC, при необходимости создаёт `PendingAction` вместо прямой мутации.
6. UI обновляется через Turbo / ответ API.

## Обоснование

- Локальный Ollama снижает утечку данных и зависимость от облачных LLM quota (см. `docs/stack/ollama.md`, `docs/stack/whisper.md`).
- Явный слой маршрутизации намерений отделяет «магию LLM» от доменной логики и аудита.

## Последствия

**Плюсы:** тестируемость IntentRouter, повторное использование для текстового ввода.

**Минусы:** задержка цепочки ASR+LLM; нужны таймауты, джобы и деградация при сбоях.

## Безопасность

Никаких `eval` или динамических вызовов по строке из модели. Только зарегистрированные обработчики intent.

## Альтернативы

- **Только облачный LLM** — отклонено как дефолт из-за ПДн и стоимости.
- **Прямое сопоставление ключевых слов без LLM** — недостаточно для естественного языка.
