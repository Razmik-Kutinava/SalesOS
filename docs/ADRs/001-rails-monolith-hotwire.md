# ADR 001: Rails-монолит и Hotwire как основной UI

**Статус:** Принято  
**Дата:** 2026-03-28  
**Контекст:** SalesOS / Omni-Agent, Phase 1–2

## Контекст

Нужна быстрая итерация для B2B CRM-lite с голосом, Telegram и будущими интеграциями. Варианты: отдельный SPA + API, микросервисы, или монолит Rails с server-rendered UI.

## Решение

Один репозиторий **Rails 8** с **Hotwire (Turbo + Stimulus)** как основным веб-клиентом. JSON API добавляется точечно (`/api/v1`) по мере необходимости интеграций и мобильных клиентов.

## Обоснование

- Меньше moving parts на старте: одна кодовая база, одна модель данных, общие политики авторизации.
- Turbo Streams дают «достаточно реактивности» для approve bar и обновлений после голосовых команд без React-стека.
- Команда и документация в `docs/stack/rails.md`, `docs/stack/hotwire-turbo-stimulus.md` уже завязаны на этот путь.

## Последствия

**Плюсы:** скорость разработки, простой деплой (один образ), естественный SSR и SEO при публичных страницах.

**Минусы:** при очень сложном UI может понадобиться больше Stimulus/турбо-логики или вынос виджета в отдельный фронт позже.

## Альтернативы

- **SPA (React/Vue) + Rails API** — отклонено для Phase 1 из-за удвоения контрактов и форм.
- **Микросервисы** — отклонено до доказанной нагрузки и границ домена.
