# ADR 002: PostgreSQL как система записей; pgvector по флагу

**Статус:** Принято  
**Дата:** 2026-03-28  
**Контекст:** CRM-данные, аудит, RAG в Phase 2

## Контекст

Требуется надёжное ACID-хранилище для лидов, задач, событий, импортов и будущего векторного поиска по документам. Альтернативы: MySQL, SQLite-only, отдельная векторная БД.

## Решение

**PostgreSQL** — единственный источник истины приложения. Расширение **pgvector** подключается на окружениях, где включён RAG; на минимальных инсталляциях Phase 1 можно обойтись без векторов.

## Обоснование

- Зрелая экосистема, JSONB для гибких полей, полнотекст и `pg_trgm` для поиска.
- Один операционный контур: бэкапы, миграции, мониторинг — см. `docs/stack/postgresql.md`, `docs/DATA-MODEL.md`.
- pgvector позволяет хранить эмбеддинги рядом с метаданными чанков без синхронизации двух СУБД на ранних этапах.

## Последствия

**Плюсы:** упрощение архитектуры, транзакционность при записи чанков и лидов.

**Минусы:** векторный поиск на очень больших объёмах может потребовать выделенного поискового движка позже; это не блокер MVP.

## Альтернативы

- **Отдельная vector DB (Qdrant, Pinecone)** — отложено до метрик объёма и latency.
- **SQLite** — не выбран для продакшен multi-user B2B по умолчанию.

## Миграции и DBA

Крупные изменения схемы согласовывать с `docs/devdep/07-dba-i-analitika-dannyh.md`; индексы `CONCURRENTLY` в проде.
