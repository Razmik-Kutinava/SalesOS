# ADR 004: Фоновые задачи — Solid Queue по умолчанию; Sidekiq при необходимости

**Статус:** Принято  
**Дата:** 2026-03-28  
**Контекст:** ASR, синк CRM, тяжёлый RAG, почта

## Контекст

Часть работы (транскрипция, вызовы Ollama при высокой нагрузке, импорт, внешние API) не должна блокировать HTTP-запрос. Нужен брокер очередей.

## Решение

По умолчанию использовать **Solid Queue** (Rails 8), опирающийся на PostgreSQL, чтобы не вводить Redis на минимальной инсталляции.

Если появятся требования к throughput, отдельным воркерам или экосистеме Sidekiq — допускается миграция или параллельное использование **Sidekiq + Redis** после отдельной оценки нагрузки (обновить этот ADR).

## Обоснование

- Меньше инфраструктурных компонентов на старте.
- Транзакционная согласованность с БД проще для idempotent jobs.
- Документация стека: `docs/stack/grouped-background-jobs-redis-solid-cable-cache.md`.

## Последствия

**Плюсы:** простой деплой, единый backup БД покрывает и очередь (для Solid Queue).

**Минусы:** при экстремальной нагрузке Redis-брокер может дать лучшую изоляцию; мониторинг очереди настроить явно.

## Операции

См. `docs/RUNBOOK.md` — разделы про рост очереди и failed jobs.

## Альтернативы

- **Только Sidekiq** — отклонено как обязательное требование Phase 1.
- **Delayed Job** — не выбран в пользу направления Rails 8 defaults.
