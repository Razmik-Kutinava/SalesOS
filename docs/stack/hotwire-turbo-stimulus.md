# Hotwire: Turbo и Stimulus

## 1. Назначение Hotwire

**Hotwire** (HTML over the wire) — подход к интерактивным веб-приложениям без обязательного тяжёлого SPA-фреймворка. Состоит из **Turbo** (навигация и обновление фрагментов страницы) и **Stimulus** (лёгкий JavaScript для поведения на DOM). Разработан в экосистеме **Basecamp** и интегрирован в **Ruby on Rails** по умолчанию.

## 2. Проблема, которую решает

Классические multi-page приложения перезагружают всю страницу при каждом клике — медленно и рвёт контекст. Полноценные SPA (React/Vue) добавляют сложность сборки, состояния и API. Hotwire предлагает **серверный рендер HTML** с **частичными обновлениями**, сохраняя простоту Rails.

## 3. Turbo Drive

Перехватывает клики по ссылкам и отправку форм, загружает следующую страницу по XHR/Fetch и **заменяет** `<body>` или использует кэш мгновенного превью (snapshot). URL в адресной строке обновляется. Снижает полные перезагрузки без изменения серверной модели маршрутов.

## 4. Turbo Frames

Страница делится на изолированные **`<turbo-frame id="...">`**. Ответ сервера может содержать только соответствующий frame — обновляется **часть** UI (например, форма с ошибками, боковая панель). Атрибуты `target`, `data-turbo-frame` управляют навигацией внутри фрейма.

## 5. Turbo Streams

Поток **операций** над DOM: `append`, `prepend`, `replace`, `remove`, `update`, `before`, `after`. Сервер отвечает MIME-типом `text/vnd.turbo-stream.html` с несколькими `<turbo-stream action="..." target="...">`. Используется после POST для обновления списков без полной перерисовки.

## 6. Action Cable и Turbo Streams

Трансляция стримов с сервера по WebSocket — обновление UI у многих клиентов (чат, статусы, счётчики).

## 7. Stimulus: философия

**Минимальный JS**: контроллеры привязываются к DOM через `data-controller`, `data-action`, `data-*-target`. Нет виртуального DOM; работа с реальным HTML, отрендеренным сервером.

## 8. Stimulus контроллер

Класс наследует `Controller`, свойства `targets`, `values`, `classes`, методы как **экшены** от событий (`click->controller#submit`). Жизненный цикл `connect`/`disconnect` для подписок.

## 9. Сочетание Turbo + Stimulus

Turbo доставляет HTML; Stimulus добавляет интерактив (раскрытие панелей, автокомплит, клиентская валидация UX). После Turbo-навигации Stimulus переподключает контроллеры к новому DOM.

## 10. Совместимость с importmap

Rails 7+ использует **importmap-rails** для загрузки Stimulus и Turbo как ES-модулей без Webpack по умолчанию. При необходимости — jsbundling для npm-пакетов.

## 11. Доступность

Сохранение семантики ссылок и форм улучшает базовую доступность по сравнению с некоторыми SPA; всё равно нужны ARIA и фокус-менеджмент для сложных виджетов.

## 12. SEO

Серверный HTML индексируется проще, чем чистый клиентский рендер (при отсутствии искусственных барьеров).

## 13. Кэширование и состояние

Turbo Drive кэширует страницы; чувствительные формы помечают `data-turbo-temporary` или отключают кэш. Состояние в `session`/`localStorage` при необходимости синхронизируется со Stimulus values.

## 14. Ограничения

Очень сложные клиентские интерфейсы (canvas, realtime графики) могут потребовать большего JS или гибрида с React-островами.

## 15. Тестирование

System-тесты Capybara видят итоговый DOM; проще отлаживать, чем полный SPA, но асинхронные стримы требуют ожиданий.

## 16. Безопасность

CSRF-токены в формах сохраняются; Turbo отправляет их как обычный браузер. Не вставлять неэкранированный пользовательский HTML в стримы.

## 17. Версии

Пакеты `@hotwired/turbo` и `@hotwired/stimulus` развиваются независимо; следить за changelog при апгрейде.

## 18. Альтернативы в экосистеме

HTMX — схожая идея атрибутов; LiveView (Phoenix) — другой стек; классический SPA — другая кривая сложности.

## 19. Производительность

Меньше JS на первую загрузку относительно крупных SPA-бандлов; размер HTML-пartial может расти — мониторить payload.

## 20. Заключение

Hotwire — практичный путь к «живому» UI в Rails с **сервером как источником истины** для разметки и **тонким слоем** поведения на Stimulus.
