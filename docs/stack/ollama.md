# Ollama: локальный рантайм для LLM

## 1. Назначение

**Ollama** — платформа для запуска **больших языковых моделей** (LLM) локально на CPU/GPU. Упрощает загрузку весов, REST API и CLI для инференса без обязательной облачной подписки. Подходит для прототипирования, приватных данных, офлайн-сценариев и снижения переменных затрат на токены.

## 2. Архитектура на высоком уровне

Демон Ollama управляет **моделями** в формате **Modelfile** (аналог Dockerfile для моделей): базовый образ весов, параметры, системный промпт, шаблон чата. Клиенты обращаются по **HTTP API** (совместимому с частью OpenAI-стиля в новых версиях) или через CLI `ollama run`.

## 3. Установка и модели

Устанавливается бинарником под ОС; модели подтягиваются командой `ollama pull <name>`. Каталог кэша весов обычно в домашней директории пользователя или в конфигурируемом пути. Размер моделей — от единиц до десятков гигабайт.

## 4. Аппаратные требования

**GPU** (NVIDIA CUDA, Apple Metal, ROCm в зависимости от платформы) сильно ускоряет инференс. На **CPU** малые модели работают терпимо для низкой конкуренции запросов; большие — с высокой латентностью. **RAM** должна вмещать модель и контекст; для 7B–8B квантованных моделей ориентир — 8–16 ГБ и выше.

## 5. Квантование

Модели поставляются в **квантованных** форматах (Q4, Q5, Q8 и др.) для уменьшения памяти и ускорения с небольшой потерей качества. Выбор варианта — компромисс качество/скорость/VRAM.

## 6. Контекстное окно

Каждая модель имеет лимит **контекстных токенов** (например, 4k, 8k, 32k). Превышение обрезается или отклоняется в зависимости от рантайма. Для RAG важно укладывать ретривнутые чанки в доступный контекст.

## 7. API инференса

Типичные эндпоинты: генерация дополнения (`/api/generate`), чат (`/api/chat`), список моделей (`/api/tags`). Параметры: `temperature`, `top_p`, `num_predict`/`num_ctx` (зависит от версии), `stop` sequences.

## 8. Шаблоны промптов

Modelfile задаёт, как **системный**, **user** и **assistant** сегменты склеиваются в сырой промпт для конкретной архитектуры (Llama, Mistral, Gemma и т.д.). Несоответствие шаблона модели ухудшает качество.

## 9. Безопасность

Локальный сервер по умолчанию слушает интерфейс; в продакшене ограничивают **bind**, **firewall**, **TLS** и аутентификацию на reverse proxy. Модель не должна быть выставлена в интернет без контроля.

## 10. Обновления моделей

Модели обновляются повторным `pull`; версии тегов (`:latest`, конкретные билды) фиксируют воспроизводимость.

## 11. Параллельные запросы

Один процесс Ollama обрабатывает запросы с очередью; под высокую конкуренцию масштабируют **несколько инстансов** за балансировщиком или выносят инференс на специализированный сервер.

## 12. Интеграция с приложениями

HTTP-клиент из Ruby/Python/JS отправляет JSON и получает потоковый или цельный ответ (**streaming** снижает воспринимаемую задержку). Таймауты и retry на стороне клиента обязательны.

## 13. Сравнение с облачными API

Плюсы Ollama: данные не покидают периметр, предсказуемая стоимость при стабильной нагрузке, отсутствие rate limit провайдера. Минусы: железо, обслуживание, задержка на слабом CPU, отставание от самых новых флагманских моделей.

## 14. Эмбеддинги

Некоторые модели и команды поддерживают **эмбеддинги** для RAG; размерность и формат должны совпадать с векторным хранилищем (pgvector и др.).

## 15. Ограничения качества

Локальные модели меньшего размера **галлюцинируют** и слабее на редких языках и узких доменах без RAG. Критик/проверка фактов и извлечение из базы знаний снижают риск.

## 16. Лицензии моделей

Веса распространяются под лицензиями конкретных моделей (Llama Community License, Apache 2.0 и др.) — проверять для коммерческого использования.

## 17. Наблюдаемость

Логирование запросов, длительности, размера промпта; метрики GPU utilization при нагрузочном тестировании.

## 18. Отладка

Повтор воспроизводимого промпта через CLI; снижение `temperature` для детерминированности; сравнение с другой моделью для изоляции проблемы.

## 19. Альтернативы в экосистеме

**llama.cpp**, **vLLM**, **text-generation-inference**, облако **OpenAI/Anthropic** — другие точки на спектре «самохост vs облако».

## 20. Заключение

Ollama — практичная оболочка для локального LLM-инференса с низким порогом входа. Успех в продакшене зависит от подбора модели, железа, шаблонов промптов и интеграции с безопасностью и наблюдаемостью.

## Приложение: переменные окружения (ориентир)

Часто настраивают `OLLAMA_HOST`, `OLLAMA_MODELS`, параметры CUDA/ROCm через стандартные переменные драйвера. Конкретный список — в документации версии.

---

## Приложение: Modelfile директивы (концептуально)

- `FROM` — базовая модель.
- `PARAMETER` — температурные и прочие параметры по умолчанию.
- `SYSTEM` — системный промпт.
- `TEMPLATE` — шаблон сообщений.

---

## Приложение: стриминг

Клиент читает **chunk**-и по мере генерации; UI показывает текст прогрессивно. На сервере приложений важно не буферизовать весь ответ до конца без необходимости.

---

## Приложение: токенизация

Длина промпта в байтах не равна токенам; для лимитов используют токенизатор модели или эвристики.

---

## Приложение: batch

Пакетная обработка нескольких промптов может быть не оптимизирована так же, как в специализированных inference-серверах — измерять throughput.

---

## Приложение: мультимодальность

Отдельные модели поддерживают изображения; требования к API и памяти выше.

---

## Приложение: fine-tuning

Ollama ориентирован на **инференс** готовых весов; дообучение выполняется внешними пайплайнами с последующим импортом совместимого формата.

---

## Приложение: сеть Docker

Контейнер приложения обращается к хосту через `host.docker.internal` или общую docker-сеть; порты публикуются осознанно.

---

## Приложение: отказоустойчивость

При падении демона клиент получает 5xx; circuit breaker и fallback (очередь, деградация функции) улучшают UX.

---

## Приложение: стоимость владения

TCO = электричество + амортизация GPU + время инженера; сравнение с облаком зависит от объёма токенов в месяц.

---

## Приложение: приватность

Данные не отправляются третьим лицам при локальном вызове; исключение — если клиент сам форвардит логи или телеметрию.

---

## Приложение: совместимость OpenAI API

В новых релизах часть клиентов переключается одной сменой base URL и ключа-заглушки — упрощает миграцию кода.

## Дополнение: tool calling / function calling

Поддержка вызова инструментов зависит от модели и версии Ollama; схемы JSON для структурированного вывода — паттерн интеграции с приложением.

## Дополнение: контекстный кэш (будущее)

Отраслевые тренды кэширования префикса KV-cache; локальные рантаймы постепенно подхватывают оптимизации.

## Дополнение: квантование AWQ / GPTQ

Разные схемы сжатия весов влияют на качество; Ollama абстрагирует через готовые blobs.

## Дополнение: model cards

Читать карточку модели: языки, лимиты коммерческого использования, benchmark на задачах близких к продукту.

## Дополнение: hybrid CPU+GPU

Некоторые конфигурации разгружают часть графа на CPU при нехватке VRAM.

## Дополнение: сетевые политики

Корпоративный egress whitelist должен включать только необходимые хосты при `pull` моделей.

## Дополнение: keep_alive и длинные сессии

HTTP keep-alive снижает оверхед при серии запросов к локальному API.

## Дополнение: логирование промптов

Отключать в продакшене или маскировать PII при централизованном логировании.

## Дополнение: сравнение с llama.cpp напрямую

Ollama — удобная обёртка; чистый llama.cpp даёт тонкий контроль билда под железо.

## Дополнение: шаблоны чата для разных семейств

ChatML, Zephyr, Llama-3 — несовместимы; Modelfile должен соответствовать весам.
