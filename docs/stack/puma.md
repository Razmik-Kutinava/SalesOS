# Puma: HTTP-сервер приложений Ruby

## 1. Назначение

**Puma** — многопоточный **application server** для Rack-приложений (Rails, Sinatra и др.). Принимает HTTP от reverse proxy (Nginx, Caddy, AWS ALB) или напрямую в dev, распределяет запросы по **воркерам** и **потокам**.

## 2. Модель процессов

Типичная конфигурация: **несколько воркеров** (fork процессов) для обхода GVL Ruby на CPU-bound и изоляции утечек памяти; внутри воркера — **пул потоков** для I/O-bound запросов.

## 3. Rack

Puma реализует интерфейс Rack: вызывает `config.ru` или Rails, передаёт env-хеш, получает статус, заголовки, тело ответа.

## 4. UNIX-сокет vs TCP

В продакшене часто слушают **UNIX domain socket** за Nginx для меньшей латентности; в контейнерах — TCP порт.

## 5. Настройка

`config/puma.rb`: `workers`, `threads`, `preload_app!`, `on_worker_boot` для переподключения к БД после fork, `bind`, `pidfile`, `environment`.

## 6. Preload

`preload_app!` загружает приложение до fork — экономия памяти (copy-on-write), но требует корректного закрытия соединений в master.

## 7. Graceful restart

**SIGUSR2** / phased restart — замена воркеров без простоя; важно для zero-downtime деплоя с правильным process manager.

## 8. Таймауты

`persistent_timeout`, `first_data_timeout` — защита от медленных клиентов.

## 9. Лимиты

Ограничение `queue_requests` и размер тела запроса на уровне Puma/proxy — защита от DoS.

## 10. Thruster / другие фронты

В Rails 8 **Thruster** может стоять перед Puma как компрессия и кэш; Puma остаётся Rack-сервером приложения.

## 11. Мониторинг

Метрики: очередь запросов, активные потоки, время обработки; интеграция с Prometheus через плагины или внешний агент.

## 12. Сравнение с Unicorn

Unicorn — только процессы, без потоков Ruby; Puma гибче для смешанной нагрузки.

## 13. Сравнение с Falcon

Falcon на async I/O — другая модель; Puma проще и де-факто стандарт Rails.

## 14. Windows

Ограниченная поддержка; продакшен обычно Linux.

## 15. Лицензия

BSD — свободное использование.

## 16. Ошибки воркера

Перезапуск упавшего воркера; логирование stack trace; корреляция с OOM.

## 17. Безопасность

Не публиковать Puma в интернет без proxy; TLS терминация на edge.

## 18. Кластер и масштабирование

Горизонтально — больше инстансов за балансировщиком; sticky sessions только если сессия в памяти (лучше cookie/Redis/БД).

## 19. Заключение

Puma — рабочая лошадка Rack; правильные `workers`/`threads` и связка с proxy критичны для стабильности под нагрузкой.

## Приложение: формула потоков

Ориентир: `threads` ≈ число одновременных I/O запросов на воркер; `workers` ≈ CPU cores для CPU-bound или чуть больше при I/O — измерять под профилем приложения.

---

## Приложение: database connections

Каждый воркер держит пул Active Record; суммарные соединения = воркеры × pool size — не превышать лимиты PostgreSQL.

---

## Приложение: systemd

Unit-файл с `Type=notify` или `simple`, `Restart=always`, лимиты `LimitNOFILE`.

---

## Приложение: Docker

Один процесс Puma на контейнер; масштабирование репликами контейнера.

---

## Приложение: early hints

Экспериментальная поддержка HTTP 103 для предзагрузки ресурсов — зависит от версии и клиентов.

---

## Приложение: логирование

Access log формат настраивается; структурированные JSON-логи через middleware Rails.

---

## Приложение: health

Отдельный лёгкий endpoint в Rails для readiness без тяжёлых запросов через Puma.

`umask` и права на socket для Nginx user.

Несколько Puma на разных сокетах/портах за одним Nginx с `server_name` маршрутизацией.

## Дополнение: JRuby под Puma

Другая модель потоков JVM — отдельный тюнинг пула.

## Дополнение: настройка backlog

`tcp_backlog` в `bind` при экстремальном числе одновременных handshake.

## Дополнение: порт reuse

SO_REUSEPORT на Linux для нескольких процессов на одном порту (редко в Rails).

## Дополнение: интеграция с Kubernetes

Deployment с readiness на HTTP endpoint Puma после старта воркеров.

Middleware ограничения длительности запроса совместно с таймаутами Puma/proxy.

Экспериментальная выдача подсказок клиенту для параллельной загрузки ресурсов.
