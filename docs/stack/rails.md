# Ruby on Rails: веб-фреймворк

## 1. Назначение

**Ruby on Rails** (Rails) — полнофункциональный серверный веб-фреймворк на Ruby, реализующий паттерн **MVC** (Model–View–Controller) и набор соглашений, ускоряющих разработку типичных веб-приложений: CRUD, формы, маршрутизация, работа с БД, фоновые задачи (через интеграции), безопасность, интернационализация.

## 2. Философия

- **Convention over configuration** — разумные умолчания уменьшают объём конфигурации.
- **DRY** — избегание дублирования через абстракции Active Record, хелперы, partials.
- **REST** — ресурсно-ориентированная маршрутизация как базовый стиль API и HTML.

## 3. Архитектура MVC

- **Model** — доменные сущности, валидации, связи, колбэки; обычно наследуют `ApplicationRecord` и маппятся на таблицы СУБД.
- **View** — шаблоны (ERB и др.), partials, хелперы; во многих проектах часть UI заменена Hotwire.
- **Controller** — обработка HTTP-запроса, выбор ответа, фильтры `before_action`, защита от CSRF для HTML-форм.

## 4. Маршрутизация

`config/routes.rb` объявляет пути, HTTP-глаголы, вложенные ресурсы, namespaces, constraints. Именованные хелперы (`_path`, `_url`) генерируются для ссылок и редиректов.

## 5. Active Record

ORM-слой:

- миграции схемы (`db/migrate`);
- ассоциации (`belongs_to`, `has_many`, `has_many :through`);
- валидации (`validates`);
- колбэки жизненного цикла;
- scope и запросы (в т.ч. безопасные при использовании плейсхолдеров);
- транзакции `ActiveRecord::Base.transaction`.

Важно избегать **SQL-инъекций**: не интерполировать пользовательский ввод в сырой SQL; использовать хеши условий и `sanitize_sql`-подобные API.

## 6. Active Model и валидации вне БД

Объекты форм и DTO могут включать `ActiveModel::Model` для единообразных валидаций без таблицы.

## 7. Action Pack (Controller + Dispatch)

Обработка запроса: роутинг, параметры `params`, strong parameters для массового присваивания, статусы ответа, `render`, `redirect_to`, кэширование фрагментов, ETag.

## 8. Action View

Шаблоны, layouts, хелперы, формы (`form_with`), CSRF-токены для session-based HTML.

## 9. Active Support

Расширения ядра Ruby: `String#blank?`, `Time`/`Date` утилиты, `Hash#deep_merge`, инструменты для колбэков и `Concern` (`ActiveSupport::Concern`).

## 10. Концерн (Concern)

Модуль с изолированным поведением, подключаемый в модели или контроллеры для переиспользования без глубокой иерархии наследования.

## 11. Безопасность (встроенные механизмы)

- **CSRF** для cookie-сессий в HTML.
- **Content Security Policy** (настраивается).
- Защита от **mass assignment** через strong parameters.
- Встроенные рекомендации по заголовкам и сессиям в гайдах безопасности Rails.

## 12. Кэширование

Уровни: фрагменты view, Russian doll caching, low-level `Rails.cache`, HTTP-кэширование. В Rails 8 усилена интеграция с **Solid Cache** как адаптером на БД.

## 13. Active Job

Абстракция фоновых задач: единый API, адаптеры к Sidekiq, Good Job, Resque и др. Позволяет откладывать почту, тяжёлые вычисления, интеграции.

## 14. Action Mailer

Генерация писем как «мейлеры» с шаблонами HTML/text, доставка через SMTP/API адаптеры, preview в development.

## 15. Action Cable

WebSockets: каналы, подписки, трансляция из моделей или сервисов. В Rails 8 возможен адаптер **Solid Cable** на БД.

## 16. Active Storage

Прикрепление файлов к моделям, direct upload, варианты изображений, облачные бэкенды (S3-совместимые и др.).

## 17. Internationalization (I18n)

Файлы локалей YAML, интерполяция, множественные формы, локализация дат и ошибок валидации.

## 18. Генераторы и задачи

`rails generate` для моделей, контроллеров, миграций. **Rake** и **Rails commands** в `bin/rails` для обслуживания БД, маршрутов, консоли.

## 19. Консоль и отладка

`rails console` — IRB с загруженным приложением. **debug** gem для breakpoints. Логи в `log/`.

## 20. Окружения

`development`, `test`, `production` — разные настройки кэша, логирования, eager load классов.

## 21. Ассеты

Классический **Sprockets** уступает место **Propshaft** + **importmap** в современном стеке; **jsbundling/cssbundling** при необходимости Node. **Hotwire** (Turbo, Stimulus) — рекомендуемый фронт без тяжёлого SPA.

## 22. Версии Rails

Мажорные релизы приносят новые дефолты (например, Solid* компоненты, изменения загрузки Zeitwerk). Минорные — улучшения и исправления. Следует читать upgrade guide при скачке мажора.

## 23. Zeitwerk

Автозагрузка кода по соглашению о путях файлов; требует согласования имён констант и файловых путей.

## 24. API-only режим

Урезанный стек без view по умолчанию — для JSON API и микросервисов на Rails.

## 25. Производительность

Eager loading (`includes`, `preload`) против N+1; индексы БД; кэш; фоновые джобы; профилирование (rack-mini-profiler и др.).

## 26. Тестирование

Minitest или RSpec; system tests; фикстуры/фабрики; тестирование мейлеров и джобов.

## 27. Развёртывание

Традиционно: Puma + reverse proxy; контейнеры; платформы PaaS. **Kamal** — современный путь деплоя Docker без Kubernetes.

## 28. Экосистема

Гемы для авторизации (Pundit, CanCanCan), аутентификации (Devise), админок, графQL, JSON API. Важно контролировать поверхность зависимостей.

## 29. Ограничения и антипаттерны

Толстые контроллеры и модели — вынос в **service objects**, **form objects**, **query objects**. Избыток колбэков в модели усложняет тестирование — предпочитать явные сервисы.

## 30. Заключение

Rails — согласованный стек для быстрой веб-разработки с упором на соглашения, безопасные практики по умолчанию и богатую экосистему. Глубокое понимание Active Record, жизненного цикла запроса и фоновых задач критично для зрелых систем.

---

## Приложение: жизненный цикл HTTP-запроса (упрощённо)

1. Веб-сервер принимает запрос и передаёт в Rack.
2. Rails middleware (логирование, cookies, session, CSRF…).
3. Router сопоставляет путь с контроллером#action.
4. Controller читает `params`, вызывает модель, готовит `@variables`.
5. View рендерит ответ или `redirect_to`.
6. Ответ уходит клиенту.

## Дополнение: конфигурация

Файлы в `config/` — окружение, база данных, кэш, кабель, маршруты, инициализаторы. Секреты — через переменные окружения или зашифрованные учётные данные (`credentials`).

---

## Дополнение: наблюдаемость

Структурированные логи, correlation id (через middleware или гемы), интеграция с APM — практики вне ядра, но стандарт индустрии для продакшена.

## 31. Credentials и секреты

`rails credentials:edit` хранит зашифрованные YAML; ключ в `RAILS_MASTER_KEY` или переменных окружения на сервере. Альтернатива — полностью внешние секрет-менеджеры (Vault, облачные Parameter Store).

## 32. Database-agnostic vs PostgreSQL-специфика

Active Record абстрагирует SQL, но для производительности часто используют `execute` с сырым SQL или специфичные типы (jsonb, массивы, CTE) — осознанный компромисс переносимости.

## 33. Multi-database

Rails поддерживает несколько БД: реплики для чтения, шардирование на уровне конфигурации — для масштабирования и разделения нагрузок.

## 34. Solid-стек в современных релизах

**Solid Queue**, **Solid Cache**, **Solid Cable** — реализации очереди, кэша и WebSocket-адаптера поверх реляционной БД, снижающие зависимость от Redis в типовых приложениях.

## 35. Health checks и readiness

Эндпоинты проверки готовности (БД, кэш, внешние зависимости) интегрируются через middleware или отдельные контроллеры для оркестраторов и балансировщиков.

## 36. Encrypted credentials и ключи

Ротация master key, хранение в секрет-менеджере платформы, разделение credential файлов по окружениям при необходимости.

## 37. Rack middleware стек

Порядок middleware влияет на поведение запроса; кастомные middleware для трейсинга, locale, maintenance mode.

## 38. Custom error pages

`public/404.html`, динамические страницы через `exceptions_app` для брендированных ошибок.
