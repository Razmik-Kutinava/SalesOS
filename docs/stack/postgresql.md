# PostgreSQL: реляционная СУБД

## 1. Назначение

**PostgreSQL** — объектно-реляционная система управления базами данных с открытым исходным кодом. Используется как основное хранилище транзакционных данных в веб-приложениях, аналитике, геосервисах и там, где важны **ACID**, расширяемость типов и богатый SQL.

## 2. Модель данных и SQL

Данные организованы в **таблицы** со **строками** и **столбцами**. Поддерживается стандарт **SQL** с множеством расширений: CTE (`WITH`), оконные функции (`OVER`), рекурсивные запросы, `LATERAL`, массивы, JSON/JSONB, полнотекстовый поиск, диапазонные типы.

## 3. ACID и транзакции

Транзакции группируют операции в атомарный блок: **Atomicity**, **Consistency**, **Isolation**, **Durability**. Уровни изоляции: Read Committed (по умолчанию), Repeatable Read, Serializable. Выбор уровня влияет на защиту от аномалий (phantom read, write skew) и на конкуренцию.

## 4. MVCC

**Multi-Version Concurrency Control** — читатели не блокируют писателей: версии строк сохраняются до vacuum. Это снижает блокировки, но требует обслуживания (**VACUUM**, autovacuum) для предотвращения раздувания и **transaction id wraparound**.

## 5. Индексы

Типы: **B-tree** (по умолчанию), **Hash**, **GiST**, **SP-GiST**, **GIN**, **BRIN**. Частичные индексы (`WHERE`), выражения, уникальные и составные индексы. Влияют на скорость `SELECT` и стоимость `INSERT`/`UPDATE`/`DELETE`.

## 6. Ограничения целостности

**PRIMARY KEY**, **FOREIGN KEY**, **UNIQUE**, **CHECK**, **NOT NULL**. Внешние ключи с каскадами `ON DELETE`/`ON UPDATE` документируют связи на уровне БД.

## 7. Нормализация и денормализация

Классические нормальные формы уменьшают избыточность; для отчётов и чтения иногда вводят денормализацию, материализованные представления или дубли с контролем консистентности.

## 8. Представления и материализованные представления

**VIEW** — сохранённый запрос. **MATERIALIZED VIEW** — снимок данных для ускорения тяжёлых агрегаций; обновляется `REFRESH`.

## 9. Хранимые процедуры и функции

Языки: **PL/pgSQL**, SQL; возможны расширения. Логика близко к данным — плюс для целостности и производительности, минус для версионирования в приложении.

## 10. Триггеры

Срабатывают на `INSERT`/`UPDATE`/`DELETE` для аудита, денормализации, валидаций на уровне БД.

## 11. Репликация

**Физическая репликация** (streaming) — standby читает WAL. **Логическая репликация** — на уровне таблиц/публикаций. Используется для масштабирования чтения, аварийного переключения, миграций с минимальным простоем.

## 12. Резервное копирование

**pg_dump** / **pg_dumpall** — логические бэкапы. **Базовый бэкап + WAL archiving** — непрерывная архивация (PITR). Политика RTO/RPO определяет выбор.

## 13. Безопасность

Роли (`ROLE`), привилегии `GRANT`, **Row Level Security (RLS)** для политик на уровне строк. SSL/TLS к серверу. Аудит через расшения или логирование.

## 14. Расширения

**CREATE EXTENSION** подключает модули: **PostGIS** (геоданные), **pgvector** (векторы), **uuid-ossp** или встроенный `gen_random_uuid()`, **pg_stat_statements** (статистика запросов), **citext** (регистронезависимый текст).

## 15. JSONB

Бинарный JSON с индексами GIN (`jsonb_path_ops`, `jsonb_ops`). Удобен для полуструктурированных полей; важно не злоупотреблять вместо нормализованной схемы без причины.

## 16. Полнотекстовый поиск

`tsvector` / `tsquery`, конфигурации языков, ранжирование `ts_rank`. Альтернатива внешним поисковым движкам для средних объёмов.

## 17. Партиционирование

Декларативное партиционирование по диапазону, списку, хешу. Улучшает обслуживание больших таблиц (архивация партиций, локальные индексы).

## 18. Производительность

**EXPLAIN (ANALYZE)** — план запроса и фактические тайминги. Настройка `shared_buffers`, `work_mem`, `maintenance_work_mem`, параллельных воркеров. Мониторинг блокировок, долгих запросов, **bloat** таблиц.

## 19. Версионирование

Мажорные релизы PostgreSQL ежегодно; проверять совместимость расширений и поведение депрекейтов. Минорные — исправления безопасности и багов.

## 20. Клиенты и драйверы

**psql** — CLI. Драйверы для Ruby (`pg`), Python, Go, Java и др. Пул соединений на стороне приложения (PgBouncer) для высокой конкуренции коротких запросов.

## 21. Миграции схемы

В приложениях схема эволюционирует через миграции; опасные операции (`ADD COLUMN` с дефолтом на большой таблице) требуют стратегий **online migration** (добавить nullable → backfill → constraint).

## 22. Локали и сортировка

`LC_COLLATE`, `LC_CTYPE` задаются при создании кластера; изменение сложно — планировать заранее.

## 23. Ограничения продукта

Не распределённое шардирование из коробки на уровне одного кластера (шардирование — на уровне приложения или внешние решения). Для экстремальных OLAP-нагрузок часто смотрят специализированные колоночные СУБД.

## 24. Сообщество и лицензия

Лицензия PostgreSQL (пермиссивная). Активная документация на **postgresql.org**.

## 25. Заключение

PostgreSQL — надёжный выбор для транзакционных приложений с богатым SQL, расширениями и зрелой эксплуатационной практикой. Понимание MVCC, индексов, изоляции и vacuum критично для стабильной работы под нагрузкой.

## Дополнение: блокировки

Уровни: строковые vs таблицы; режимы `ACCESS SHARE` … `ACCESS EXCLUSIVE`. Deadlock detection — автоматический откат одной транзакции. Дизайн транзакций короткими и в фиксированном порядке доступа к ресурсам снижает взаимные блокировки.

---

## Дополнение: connection pooling

Без пула каждый воркер приложения открывает соединение — риск исчерпания `max_connections`. **PgBouncer** в режимах session/transaction/statement — стандартное решение.

## Дополнение: последовательности и SERIAL

`SERIAL` / `BIGSERIAL` — синтаксический сахар над последовательностями (`SEQUENCE`). При массовых вставках и репликации важно учитывать сдвиги и кэширование sequence.

## Дополнение: временные таблицы

`CREATE TEMP TABLE` — сессионные таблицы для ETL и сложных отчётов без постоянной схемы.

## Дополнение: LISTEN / NOTIFY

Асинхронные уведомления между сессиями — лёгкий механизм сигналинга; не замена полноценной очереди сообщений при высокой нагрузке.

## Дополнение: FDW (Foreign Data Wrapper)

Доступ к внешним источникам как к таблицам PostgreSQL — интеграция с другими СУБД и файлами через расширения.

## Дополнение: статистика планировщика

`ANALYZE` обновляет статистику для оптимизатора; после крупных изменений данных может потребоваться вручную или по расписанию.

## Дополнение: fillfactor и физическое хранение

`FILLFACTOR` на индексах и таблицах оставляет место под hot updates и снижает фрагментацию страниц в write-heavy сценариях.

## Дополнение: оконные функции (примеры применения)

`ROW_NUMBER()`, `RANK()`, `LAG()`/`LEAD()` для аналитики без подзапросов; партиционирование `PARTITION BY` для метрик по сегментам.

## Дополнение: UPSERT

`INSERT ... ON CONFLICT DO UPDATE` — идемпотентная запись и синхронизация staging-данных.

## Дополнение: advisory locks

`pg_advisory_lock` для координации между воркерами приложения без отдельной таблицы mutex.

## Дополнение: COPY

Массовая загрузка CSV/TEXT в таблицу быстрее множественных INSERT для миграций данных.

## Дополнение: pg_stat_statements

Нормализованный текст запросов, total_time, calls — основа оптимизации самых дорогих запросов.

## Дополнение: autovacuum tuning

Параметры `autovacuum_vacuum_scale_factor` и пороги для крупных таблиц — баланс между bloat и IO.

## Дополнение: логические декодеры

Поток изменений для CDC в Kafka/другие системы через плагины.

## Дополнение: секционирование по времени

Ежемесячные партиции для логов и событий с автоматическим созданием job-ом.

## Дополнение: work_mem и сортировки

Временные файлы на диске при превышении work_mem — симптом тяжёлых сортировок в explain.

## Дополнение: parallel query

`max_parallel_workers_per_gather` ускоряет крупные сканы при достаточных ресурсах.
