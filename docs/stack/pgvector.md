# pgvector: векторное расширение PostgreSQL

## 1. Назначение

**pgvector** — расширение для PostgreSQL, добавляющее тип данных **вектор** и операции **похожести** (similarity search) для числовых представлений (эмбеддингов) текста, изображений, аудио и других объектов. Типичное применение: **семантический поиск**, **RAG** (retrieval-augmented generation), рекомендательные системы и дедупликация по смыслу.

## 2. Установка и подключение

Расширение устанавливается на уровне сервера PostgreSQL (пакет ОС, образ Docker или сборка из исходников). В базе выполняется:

```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

Требуется совместимая версия расширения с мажорной версией PostgreSQL.

## 3. Тип `vector`

Столбец объявляется с фиксированной **размерностью** (число измерений эмбеддинга), например `vector(1536)` для популярных моделей. Вектор хранится как массив чисел с плавающей точкой; точность и формат должны соответствовать модели, генерирующей эмбеддинги.

## 4. Метрики расстояния

Поддерживаются операторы:

- **L2** (евклидово расстояние) `<->`
- **косинусное** расстояние `<=>` (для нормализованных векторов часто эквивалентно углу)
- **внутреннее произведение** `<#>` (inner product), полезно когда длины нормализованы и знак важен

Выбор метрики должен **совпадать** с тем, как модель и пайплайн индексации были обучены/нормализованы.

## 5. Индексы для ускорения поиска

Для больших таблиц линейный перебор (exact search) становится дорогим. pgvector поддерживает приближённые индексы:

- **IVFFlat** — кластеризация в списки (lists), быстрый ANN при настройке `lists` и `probes`.
- **HNSW** (в поддерживаемых версиях) — граф соседей, часто даёт лучший recall/latency trade-off при больших объёмах.

Параметры строительства индекса (`lists`, `m`, `ef_construction` для HNSW) влияют на время построения, размер на диске и качество поиска.

## 6. Точный vs приближённый поиск

**Exact** (`ORDER BY embedding <-> query LIMIT k`) — гарантированно найдёт ближайших, но O(n) по строкам без эффективного индекса или с seq scan.

**Approximate** — использует индекс; возможен пропуск истинно ближайших соседей (ниже recall). Для RAG обычно допустим компромисс при мониторинге качества ответов.

## 7. Нормализация векторов

Для косинусной метрики часто **L2-нормализуют** векторы перед сохранением, чтобы евклидово расстояние коррелировало с косинусной близостью. Несогласованность нормализации между обучением, индексом и запросом искажает ранжирование.

## 8. Схема хранения (паттерн)

Типичная таблица фрагментов знаний:

- `id` — первичный ключ.
- `content` — текст чанка.
- `metadata` — jsonb (источник, страница, теги).
- `embedding` — `vector(n)`.

Связь с «документом» верхнего уровня — через внешний ключ или jsonb-поле.

## 9. Чанкинг документов

RAG требует разбиения длинных документов на **чанки** фиксированного размера или по структуре (заголовки, абзацы). Перекрытие (overlap) между чанками снижает потерю контекста на границах. Размер чанка влияет на точность retrieval и стоимость эмбеддинга.

## 10. Модели эмбеддингов

Внешние API (OpenAI, Cohere и др.) или **локальные** модели (sentence-transformers, Ollama) генерируют векторы фиксированной размерности. Нельзя смешивать в одном столбце эмбеддинги от **разных** моделей без миграции и переиндексации.

## 11. Запросы с фильтрами

Комбинация векторного поиска с SQL-фильтрами (`WHERE tenant_id = … AND category = …`) — стандартный паттерн мультитенантности. Порядок применения фильтра и индекса влияет на план: иногда сначала узкий фильтр, затем вектор среди оставшихся строк.

## 12. Производительность и память

Индексы HNSW/IVFFlat занимают заметный объём RAM/диска. `maintenance_work_mem` при построении индекса увеличивают для ускорения. Мониторинг: время запроса, доля seq scan vs index scan.

## 13. Обновление и переиндексация

При смене модели эмбеддингов необходимо **пересчитать** все векторы и **перестроить** индекс. Версионирование модели в метаданных помогает избежать тихой несовместимости.

## 14. Ограничения

pgvector не заменяет полнотекстовый поиск по ключевым словам во всех сценариях: **гибридный** поиск (BM25 + vector) часто даёт лучший recall. Ограничения по размерности и нагрузке зависят от железа и настроек PostgreSQL.

## 15. Безопасность и мультитенантность

Изоляция данных на уровне приложения обязательна: векторный поиск без `WHERE` может вернуть чужие фрагменты. **Row Level Security** в PostgreSQL совместим с обычными запросами к таблицам с `vector`.

## 16. Мониторинг качества RAG

Метрики вне СУБД: hit rate релевантных чанков, пользовательские оценки, LLM-as-judge — для подстройки `top_k`, порогов расстояния и размера чанка.

## 17. Альтернативы (контекст рынка)

Специализированные векторные БД (Qdrant, Milvus, Weaviate, Pinecone) оптимизированы под ANN на кластерах. pgvector выигрывает **простотой стека**, **транзакциями** и **единым движком** с бизнес-данными.

## 18. Интеграция с ORM

В Ruby **neighbor** и аналоги маппят тип `vector`; в других стеках — raw SQL или драйверы с кастомными типами. Важно экранировать параметры запросов.

## 19. Версии расширения

Между версиями pgvector меняются возможности индексов и операторы — читать release notes при апгрейде.

## 20. Заключение

pgvector превращает PostgreSQL в хранилище для семантического поиска при минимальном усложнении инфраструктуры. Успех зависит от согласованности модели эмбеддингов, метрики, индекса и прикладной логики чанкинга.

## Приложение: пример SQL (иллюстрация)

```sql
SELECT id, content, embedding <=> '[...]'::vector AS dist
FROM knowledge_chunks
ORDER BY embedding <=> '[...]'::vector
LIMIT 5;
```

(Плейсхолдер `[...]` заменяется реальным массивом чисел из приложения.)

---

## Приложение: параметры IVFFlat (концептуально)

- `lists` — число кластеров; больше lists → точнее, но тяжелее build.
- `probes` (через `SET ivfflat.probes`) — сколько списков просматривать при запросе; больше probes → выше recall, выше latency.

---

## Приложение: hybrid search (идея)

Два параллельных скоринга — полнотекст (`ts_rank`) и вектор — затем нормализация и взвешенная сумма или reranker. Реализуется в SQL или в приложении.

## Дополнение: дедупликация

Кластеризация эмбеддингов или порог близости для слияния дубликатов контента — прикладная задача поверх pgvector.

---

## Дополнение: batch insert

Массовая вставка чанков эффективнее через `COPY` или батчи в транзакции; после крупной загрузки — `ANALYZE` и пересмотр индекса.

---

## Дополнение: ограничение размерности

Слишком большая размерность без необходимости увеличивает размер строк и время расчёта; иногда применяют PCA или специализированные компактные модели.

---

## Дополнение: этика и PII

Эмбеддинги могут косвенно раскрывать свойства данных; PII в чанках нужно маскировать до индексации согласно политике.

---

## Дополнение: резервное копирование

Векторные столбцы входят в обычный бэкап PostgreSQL; при восстановлении версия расширения должна совпадать.

---

## Дополнение: параллельные запросы

Несколько одновременных ANN-запросов конкурируют за CPU и I/O; connection pooling и лимиты нагрузки защищают кластер.

---

## Дополнение: тестирование

Синтетические датасеты с известными ближайшими соседями помогают проверить recall индекса против brute force на подвыборке.

---

## Дополнение: миграция с внешней векторной БД

План: выгрузить id+vector+metadata → загрузить в PostgreSQL → переключить чтение → удалить старый сервис после валидации.

## Дополнение: halfvec и новые типы

В развитии расширения появляются оптимизации хранения (например, с пониженной точностью) — следить за changelog для экономии места.

## Дополнение: concurrent index build

`CREATE INDEX CONCURRENTLY` снижает блокировки на продакшене при добавлении HNSW/IVFFlat на живой таблице.

## Дополнение: sparse vectors (перспектива)

Индустрия движется к разреженным представлениям в отдельных системах; в pgvector основной фокус — плотные векторы.
