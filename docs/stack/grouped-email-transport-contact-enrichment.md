# Сгруппировано: транзакционная почта, провайдеры, обогащение контактов

Документ объединяет **доставку электронной почты** (SMTP и HTTP API), **трекинг вовлечённости** и **поиск корпоративных контактов** (email finder). Описание обще, без привязки к конкретному продукту.

---

## Часть 1. Модель доставки email

**SMTP** — классический протокол передачи писем на MX получателя через промежуточный relay (провайдер). **HTTP API** — REST вызов к облачному сервису, который сам доставляет через свою инфраструктуру; часто удобнее для приложений (вебхуки, шаблоны, аналитика).

**MTA** (Mail Transfer Agent) провайдера принимает от приложения и ставит в очередь исходящей доставки.

---

## Часть 2. DNS и репутация домена

**SPF** (TXT): какие серверы могут отправлять от имени домена.

**DKIM** (подпись заголовков/тела): криптографическая проверка подлинности.

**DMARC** (политика для SPF/DKIM fail): отчёты и указание обработки подделок.

Без корректных записей письма попадают в **спам** или отбрасываются.

**Выделенный поддомен** для транзакционной почты (`mail.example.com`) изолирует репутацию от корпоративной переписки.

**Прогрев домена/IP**: постепенное увеличение объёма для новых отправителей.

---

## Часть 3. SendGrid (Twilio)

Крупный провайдер: API v3, шаблоны, списки, статистика открытий/кликов, подавление списков. Планы от бесплатного лимита до enterprise.

Интеграция: gem `sendgrid-actionmailer` или сырой HTTP. **Webhooks** событий: delivered, bounce, spam report, unsubscribe.

**Безопасность**: API keys с областью видимости; ротация.

---

## Часть 4. Resend

Современный API-first провайдер с простым REST, ориентацией на разработчиков, **бесплатным tier** для старта. Поддержка доменов, React-email шаблонов в экосистеме JS (опционально), webhooks.

Подходит для транзакционных писем и маркетинговых рассылок в зависимости от политики сервиса.

---

## Часть 5. Amazon SES

Низкая цена за объём, интеграция с AWS IAM, **configuration sets**, SNS для bounce/complaint. Требует выхода из sandbox для продакшена. Сложнее первоначальная настройка, зрелая масштабируемость.

---

## Часть 6. Другие провайдеры (ориентиры)

**Mailgun**, **Postmark** (фокус на транзакционные), **Brevo** (Sendinblue), **SparkPost** — сравнивать по цене, доставляемости в целевых почтовых ящиках, SLA и GDPR.

---

## Часть 7. Action Mailer в Rails

Абстракция писем: классы `ApplicationMailer`, методы с `mail(to:, subject:)`, шаблоны `.html.erb` / `.text.erb`. Доставка синхронно или через Active Job.

Настройка `delivery_method` и credentials для SMTP/API.

---

## Часть 8. Открытия и клики (tracking)

**Tracking pixel** — прозрачное изображение 1×1 с уникальным URL: при загрузке почтовым клиентом фиксируется открытие. **Ограничения**: Apple Mail Privacy, блокировщики, plain-text клиенты — метрики занижены.

**Wrapped links** — редирект через сервер провайдера для учёта кликов.

**Этика и право**: прозрачность в политике конфиденциальности, соответствие CAN-SPAM, GDPR, местным законам о рекламе.

---

## Часть 9. Bounce и complaint

**Hard bounce** — несуществующий адрес; **soft bounce** — временная ошибка. **Complaint** — нажатие «спам» пользователем.

Автоматическое **подавление** адресов снижает порчу репутации домена.

---

## Часть 10. Шаблоны и персонализация

Подстановка имён, локалей, динамических блоков. Избегать **инъекций HTML** из пользовательского ввода — экранирование.

---

## Часть 11. Объём и лимиты

Провайдеры вводят **rate limits**; очередь на стороне приложения с backoff при 429.

---

## Часть 12. Snov.io

Сервис **поиска email** по домену компании, должности, LinkedIn (в рамках ToS продукта). API: поиск кандидатов, верификация, drip-кампании в некоторых планах.

**Верификация** снижает bounce: catch-all домены дают неопределённость.

**Кредиты** API по подписке.

---

## Часть 13. Hunter.io

Аналогичный **Email Finder** и **Verifier** по домену и имени. **Confidence score** для найденных адресов. Интеграции и API для обогащения CRM.

---

## Часть 14. Юридические и этические аспекты обогащения

Персональные данные сотрудников компаний регулируются **GDPR**, локальными законами и **политиками источников**. Использовать данные для **легитимного интереса B2B** или согласия — консультация с юристом. Не покупать «сырые» списки без происхождения.

---

## Часть 15. Архитектура пайплайна

1. Идентификатор компании (домен).
2. Запрос к API finder с rate limit.
3. Верификация (отдельный вызов или встроенный флаг).
4. Сохранение в CRM с источником и датой.
5. Исключение из рассылки при opt-out.

---

## Часть 16. Дедупликация и нормализация

Lowercase email, trim, punycode для IDN доменов, приоритет подтверждённых адресов.

---

## Часть 17. Тестирование

Песочницы провайдеров, **mail catcher** (Mailhog, Letter Opener) в development, не слать реальным пользователям из CI.

---

## Часть 18. Мониторинг доставки

Дашборды провайдера + собственные метрики: время до inbox (seed accounts), доля bounce/spam.

---

## Часть 19. Мультиязычность

`Accept-Language` не применим к email; явный выбор локали пользователя или сегмента.

---

## Часть 20. Вложения и размер

Лимиты провайдера и получателя; большие файлы через ссылки на облако.

---

## Часть 21. Заключение

Транзакционная почта требует **DNS-гигиены**, **наблюдаемости** и **уважения к подписчику**. Обогащение контактов — мощный инструмент B2B при соблюдении правил и верификации качества данных.
