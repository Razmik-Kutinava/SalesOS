# Сгруппировано: фоновые задачи, очереди, Redis, Solid Queue / Cache / Cable

Документ объединяет технологии **асинхронного выполнения**, **кэширования** и **realtime** в экосистеме Ruby on Rails и соседних стеках. Описание обще, без привязки к конкретному продукту.

---

## Часть 1. Зачем нужны фоновые задачи

Веб-запрос должен завершаться быстро для пользователя. Операции, требующие секунды и минуты — отправка тысяч писем, парсинг сайтов, вызов LLM, генерация отчётов, конвертация видео — выносятся в **фон (background jobs)**. Исполнитель забирает задачу из **очереди**, обрабатывает в отдельном процессе/потоке, результат пишет в БД или шлёт уведомление.

Преимущества: отзывчивость UI, повтор при ошибках (retry), расписание (cron-like), rate limiting вне HTTP-цикла.

---

## Часть 2. Active Job как абстракция Rails

**Active Job** предоставляет единый API: `perform_later`, очереди по имени, `retry_on`, `discard_on`, сериализация аргументов GlobalID. Конкретный **адаптер** определяет, куда кладётся задача: Sidekiq, Good Job, Resque, Delayed Job, **Solid Queue** и др.

Смена адаптера при сохранении интерфейса упрощает миграцию инфраструктуры.

---

## Часть 3. Redis

**Redis** — in-memory структура данных с персистенцией на диск (RDB/AOF). Используется как:

- брокер очередей и хранилище состояния для Sidekiq;
- кэш и session store;
- pub/sub;
- rate limiting (sliding window);
- блокировки (distributed lock с осторожностью).

Типы данных: строки, списки, множества, sorted sets, hashes, streams (лог событий), HyperLogLog, bitmaps.

**Персистенция**: снимки RDB периодически; AOF журнал команд — сильнее durability, больше диска.

**Высокая доступность**: Redis Sentinel (failover master), Redis Cluster (шардирование).

**Память**: все данные должны помещаться в RAM (с оговорками для eviction политик); мониторинг `used_memory`, фрагментации.

**Безопасность**: `requirepass`, TLS, bind на приватный интерфейс, ACL в новых версиях.

**Ограничения**: не замена реляционной БД для транзакционных отчётов; сложные JOIN — в PostgreSQL.

---

## Часть 4. Sidekiq

**Sidekiq** — популярный обработчик фоновых задач на Ruby с **многопоточностью** внутри процесса. Задачи — JSON в Redis-списках; воркеры извлекают job, десериализуют класс и аргументы, вызывают `perform`.

**Sidekiq Pro/Enterprise** добавляют batch, rate limiting, уникальность job, расширенный ретрай.

**Конкуренция**: потоки разделяют память — thread-safe код в джобах; соединения Active Record через `connection_pool`.

**Мониторинг**: Web UI, интеграция с Prometheus, dead job queue.

**Идемпотентность**: повтор после crash может выполнить job дважды — дизайн `perform` должен быть устойчив.

**Лимиты Redis**: большие payload в job плохи — передавать id записей, не гигантские JSON.

---

## Часть 5. Good Job

**Good Job** — адаптер Active Job на **PostgreSQL** как очереди (таблицы jobs), без Redis. Подходит, когда хочется **одного движка** с основной БД и проще эксплуатация для малых/средних нагрузок.

Особенности: `good_job` процесс с concurrency, cron через `config`, dashboard. Миграции создают таблицы очередей.

Сравнение с Sidekiq: меньше компонентов, но нагрузка на PostgreSQL выше при очень большом числе job/сек.

---

## Часть 6. Resque и Delayed Job (наследие)

**Resque** — Redis + отдельный процесс на воркер (fork), проще модель памяти, выше overhead.

**Delayed Job** — таблица в БД, минималистичный старт; слабее под экстремальный throughput.

Знание истории полезно при поддержке легаси.

---

## Часть 7. Solid Queue

**Solid Queue** (Rails 8+) — очередь Active Job поверх **реляционной БД** (SQLite/PostgreSQL/MySQL по конфигу). Цели: **убрать Redis** из типового стека, упростить деплой, использовать транзакции БД для атомарности постановки job вместе с бизнес-данными.

Процесс `bin/jobs` или встроенный supervisor запускает воркеры; таблицы для ready/claimed/failed jobs.

Производительность: для высокочастотных job профилировать индексы и блокировки; при пределе масштабировать горизонтально воркеры.

---

## Часть 8. Solid Cache

**Solid Cache** — адаптер `Rails.cache` на БД вместо Redis/Memcached. Записи как строки/blob с TTL и ключами. Плюс: один стек; минус: латентность выше, чем у чистого in-memory.

Подходит для фрагментного кэша, low-volume ключей, сред без Redis.

---

## Часть 9. Solid Cable

**Solid Cable** — адаптер **Action Cable** на БД: сообщения WebSocket бродкастятся через запись/чтение в таблицах (упрощённо). Снижает необходимость Redis для Cable в небольших приложениях.

Для высокого realtime-трафика Redis adapter может оставаться предпочтительнее.

---

## Часть 10. Выбор между Redis и «всё в PostgreSQL»

| Критерий | Redis + Sidekiq | Solid Queue / Good Job |
|----------|-----------------|-------------------------|
| Компоненты | +Redis | только БД |
| Пик job/сек | выше потолок | смотреть по профилю |
| Атомарность с БД | двухфазные паттерны | проще в одной транзакции |
| Операционная зрелость | очень распространено | растущая экосистема |

## Часть 11. Cron и периодические задачи

**Whenever**, **sidekiq-cron**, **good_job cron**, **Solid Queue recurring** — планирование. Альтернатива: системный cron вызывает `rails runner`.

Важно: не дублировать job при нескольких инстансах без **advisory locks** или уникальных ключей.

---

## Часть 12. Надёжность и DLQ

**Dead letter queue**: job после исчерпания retry — ручная проверка, re-enqueue, алертинг.

**Идемпотентные ключи** у провайдеров внешних API предотвращают дубликаты платежей/писем.

---

## Часть 13. Наблюдаемость

Трассировка job id в логах, метрики: длина очереди, latency, error rate, retry count. OpenTelemetry для Ruby.

---

## Часть 14. Безопасность

Аргументы job не должны содержать сырые секреты; GlobalID раскрывает существование записей — учитывать авторизацию при десериализации в edge-кейсах.

---

## Часть 15. Масштабирование

Вертикально: больше воркеров/потоков с учётом CPU и пула БД. Горизонтально: несколько машин с общим Redis или общей БД-очередью — избегать split-brain.

---

## Часть 16. Тестирование

`perform_enqueued_jobs` в ActiveJob test helper; отдельные интеграционные тесты адаптера на staging.

---

## Часть 17. Миграция между системами

Пауза постановки новых job, дренаж старой очереди, двойной продюсер временно — рискованно; планировать окно обслуживания.

---

## Часть 18. Заключение блока

Фон и кэш — сердце асинхронной архитектуры Rails-приложений. Выбор Redis vs DB-очереди — компромисс простоты стека и пиковой пропускной способности. Solid* семейство снижает операционную поверхность для типовых приложений.
