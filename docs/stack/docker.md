# Docker: контейнеризация приложений

## 1. Назначение

**Docker** — платформа упаковки приложений в **контейнеры**: изолированные процессы с файловой системой и сетью, построенные из **образов**. Обеспечивает воспроизводимость окружения между dev, CI и продакшеном.

## 2. Образ и контейнер

**Image** — неизменяемый слоистый снимок ФС + метаданные запуска. **Container** — запущенный экземпляр образа с writable слоем поверх. Образы версионируются тегами и хранятся в **registry** (Docker Hub, GHCR, ECR и др.).

## 3. Dockerfile

Декларативная последовательность инструкций: `FROM` базового образа, `WORKDIR`, `COPY`/`ADD`, `RUN` установка пакетов, `ENV`, `EXPOSE`, `CMD`/`ENTRYPOINT`. Каждая `RUN` создаёт слой — группировка команд снижает размер.

## 4. Слои и кэш

Повторная сборка использует кэш неизменённых слоёв. Порядок инструкций важен: редко меняющиеся шаги выше, часто меняющийся код — ниже.

## 5. Многостадийные сборки

**multi-stage build**: стадия `builder` с компиляторами, финальная `runtime` только с артефактом — меньше attack surface и размер образа.

## 6. Сеть

По умолчанию **bridge**; контейнеры в одной user-defined сети резолвят имена друг друга DNS. **host** режим на Linux убирает NAT-latency, но слабее изоляция.

## 7. Тома (volumes)

Персистентные данные вне writable слоя контейнера: **named volumes**, **bind mount** к хосту. Базы данных и логи — типичные кандидаты.

## 8. Compose

**Docker Compose** (файл `compose.yaml`) описывает несколько сервисов, сети, тома, переменные — один командой поднять стек приложения + БД + Redis.

## 9. Переменные окружения

`ENV` в Dockerfile и `environment` в Compose; секреты лучше через **Docker secrets**, внешние vault или `--env-file`, не коммитить в git.

## 10. Пользователь в контейнере

Не запускать процессы от root в продакшене при возможности: `USER` в Dockerfile с numeric UID для Kubernetes securityContext.

## 11. Здоровье контейнера

**HEALTHCHECK** в Dockerfile или orchestrator probes — отличие «процесс жив» от «приложение отвечает».

## 12. Ресурсы

Лимиты CPU/memory в Compose/Kubernetes предотвращают noisy neighbor на shared-хосте.

## 13. Безопасность

Сканирование образов (Trivy, Grype), минимальные базовые образы (**distroless**, **alpine** с оговорками musl), обновление базы.

## 14. Оркестрация

**Kubernetes**, **Nomad**, **Docker Swarm** — планирование реплик, rolling updates. **Kamal** — лёгкий деплой без полного K8s.

## 15. CI/CD

Сборка образа в pipeline, push в registry, deploy с новым digest. Кэш buildkit в CI ускоряет сборки.

## 16. Ограничения

Контейнеры разделяют ядро ОС — не полные VM. Windows/macOS Docker Desktop использует лёгкую VM под капотом.

## 17. Лицензирование Docker Desktop

На крупных корпоративных условиях проверять лицензию Desktop vs **Moby** / **Colima** / **Rancher Desktop**.

## 18. Отладка

`docker logs`, `docker exec -it` shell внутри контейнера, `docker inspect` метаданные.

## 19. Антипаттерны

Хранение данных только в контейнере без volume — потеря при удалении. Жирные образы с неиспользуемыми пакетами.

## 20. Заключение

Docker — де-факто стандарт упаковки веб-приложений; понимание слоёв, сетей, томов и безопасности базовое для эксплуатации.

## Приложение: .dockerignore

Исключение `node_modules`, `.git`, tmp из контекста сборки — ускорение и меньше утечек.

---

## Приложение: BuildKit

Современный билдер с параллельными стадиями и секретами `RUN --mount=type=secret`.

---

## Приложение: платформы

`--platform linux/amd64` при сборке на Apple Silicon для совместимости с прод-серверами.

---

## Приложение: tagging

Семантические теги + `git sha` для трассируемости деплоев.

---

## Приложение: rootless

Rootless Docker снижает риск escape из контейнера на хост.

---

## Приложение: логирование

Драйверы логов json-file, syslog, fluentd — интеграция с централизованным логированием.

---

## Приложение: time sync

Дрейф часов в VM на Mac иногда влияет на JWT — NTP на хосте.

---

## Приложение: PID 1

Правильный init-процесс или `tini` для репинга зомби-процессов в контейнере.

---

## Приложение: graceful shutdown

Обработка SIGTERM в приложении для завершения соединений перед kill.

---

## Дополнение: образы Ruby/Rails

Preinstall гемов в слое кэша, assets precompile в build, миграции на старте или отдельной джобой — предмет best practices сообщества.

## Дополнение: devcontainers

VS Code Dev Containers — разработка внутри контейнера с единым окружением команды.

## Дополнение: docker compose profiles

Профили для включения только нужных сервисов (observability, one-off tools).

## Дополнение: bake / compose buildx

Параллельная сборка нескольких образов в CI.

## Дополнение: SBOM

Software Bill of Materials после сборки для compliance.

## Дополнение: signing images

Cosign подпись образов в registry для политики admission в K8s.

## Дополнение: resource constraints в compose

`deploy.resources` (в swarm-режиме) или аналоги в v3 compose — лимиты памяти для dev-стека.

## Дополнение: Windows containers

Отдельная линейка образов; реже в Ruby-стеке.

## Дополнение: podman

CLI-совместимая альтернатива без демона rootful по умолчанию.

## Дополнение: orphan containers

`docker compose down --remove-orphans` для уборки после переименования сервисов.

## Дополнение: heredoc в Dockerfile

Синтаксис `COPY <<EOF` для встраивания конфигов без отдельных файлов в репозитории.

## Дополнение: docker scout

Сканирование уязвимостей базовых образов в CLI.

## Дополнение: лимиты логов

`json-file` log driver `max-size`/`max-file` предотвращает заполнение диска.

## Дополнение: export образа

`docker save` / `docker load` для air-gapped сред без registry.

## Дополнение: prune

`docker system prune` освобождает место от неиспользуемых слоёв после итераций CI.
