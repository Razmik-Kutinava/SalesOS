# Безопасность и приватность

**Версия:** 0.1  
**Статус:** Базовые требования и угрозы (детализация по мере аудита)

---

## 1. Область применения

Веб-приложение SalesOS, Telegram-бот, хранилища данных, интеграции (Ollama, ASR, CRM, почта).

---

## 2. Классификация данных

| Класс        | Примеры                    | Меры                          |
|-------------|----------------------------|-------------------------------|
| Публичные   | маркетинговый сайт         | обычный HTTPS                 |
| Внутренние  | стадии воронки, метаданные | authz, аудит                  |
| Конфиденциальные | ПДн: ФИО, email, телефон | минимизация, шифрование, лог-маскирование |
| Секреты     | API keys, пароли           | secret manager, rotation      |

---

## 3. ПДн (персональные данные)

- Минимизировать сбор: только нужное для CRM-процесса.
- **Цель обработки** — ведение коммерческих отношений; политика конфиденциальности на сайте/в приложении (юридический текст — вне этого документа).
- **Согласия** на рассылку — отдельный флаг при включении email-маркетинга.
- **Право на удаление** — процедура через админа + каскадное удаление/анонимизация.

---

## 4. Хранение секретов

- Не коммитить в git (`.env` в `.gitignore`).
- Прод: переменные окружения на сервере, либо Vault / cloud secret manager.
- Токены API хранить как **digest** (bcrypt) где применимо.

---

## 5. Аутентификация

- Пароли: современный алгоритм (bcrypt/argon2 через Rails).
- Защита от brute-force: rate limit на login, lockout опционально.
- Сессии: secure cookie, SameSite, HttpOnly.

---

## 6. Авторизация

- Проверка на **каждом** действии с ресурсом (policy object).
- Scope по `account_id` для мультитенантности.
- Разделение ролей: owner / admin / user.

---

## 7. Угрозы (краткая STRIDE-ориентация)

| Угроза              | Пример                         | Митигация                          |
|---------------------|--------------------------------|------------------------------------|
| Spoofing            | поддельный webhook             | секрет в URL/HMAC                  |
| Tampering           | изменение stage через API      | authz + audit                      |
| Repudiation         | отрицание действия             | `lead_events` с actor              |
| Information disclosure | утечка PII в логах        | маскирование, минимум логов        |
| DoS                 | flood голосовых upload         | rate limit, размер тела            |
| Elevation           | user → admin                   | проверка роли на сервере           |

---

## 8. LLM / Ollama

- **Не** исполнять произвольный код из ответа модели.
- Строгая **схема** intent + allow-list действий.
- Логировать hash промпта, не сырой аудио-текст в общих логах при высоком риске.
- Локальный Ollama снижает утечку данных наружу; удалённый — только TLS.

---

## 9. Логи

- Структурированные JSON логи.
- Маскировать email/телефон regex-заменой.
- Ротация и срок хранения (например 30–90 дней).

---

## 10. Аудит

- Все изменения лида через `lead_events`.
- Админ-действия (смена роли, экспорт) логировать отдельно.

---

## 11. Транспорт

- TLS 1.2+ везде в проде.
- HSTS при стабильном домене.

---

## 12. Заголовки безопасности

- CSP, X-Frame-Options, X-Content-Type-Options.
- Referrer-Policy restrictive.

---

## 13. CSRF

- Включён для cookie-based сессий на мутациях.

---

## 14. XSS

- Экранирование в ERB; доверять только санитизированному HTML если rich text.

---

## 15. SQL injection

- Active Record только параметризованные запросы; сырой SQL редко и с биндингами.

---

## 16. Файлы загрузки

- Проверка MIME/extension, лимит размера, сканирование опционально.
- Хранение вне web root.

---

## 17. Голосовые данные

- Политика: не хранить аудио после транскрипции или TTL ≤ 24h.
- Доступ к временным файлам только процессом приложения.

---

## 18. Telegram

- Валидация webhook secret.
- Не логировать полные update JSON в prod (PII).

---

## 19. Резервные копии

- Шифрование бэкапов at-rest.
- Доступ к бэкапам по принципу least privilege.

---

## 20. Инциденты

- Процедура: изоляция → ротация секретов → уведомление заинтересованных сторон по регламенту.
- Контакт ответственного (заполнить в RUNBOOK).

---

## 21. Зависимости

- `bundle audit` / GitHub Dependabot.
- Патчинг критичных CVE в срок SLA команды.

---

## 22. Доступ к продакшену

- SSH по ключам, без паролей.
- 2FA в панели хостинга/облака.

---

## 23. Минимальные права БД

- Приложение: не superuser.
- Read replica credentials только SELECT.

---

## 24. Сетевая сегментация

- БД недоступна из интернета; только app subnet.

---

## 25. ПДн в РФ

- Уточнить требования локализации данных с юристами; при необходимости хостинг в РФ.

---

## 26. ПДн в других юрисдикциях

- GDPR: DPA, SCC при трансграничной передаче.

---

## 27. Penetration test

- Периодичность: ежегодно или перед крупным релизом.

---

## 28. Security.txt

- Разместить `/.well-known/security.txt` с контактом.

---

## 29. Секреты в CI

- GitHub Actions secrets / переменные в CI; не echo в логи.

---

## 30. Code review

- Любой PR с auth/security — внимательный review.

---

## 31. API tokens

- Scope минимальный; возможность отзыва.

---

## 32. CORS

- Не `*` с credentials.

---

## 33. Subresource Integrity

- Для CDN скриптов если появятся.

---

## 34. Rate limiting

- На auth, voice upload, публичные API.

---

## 35. Account takeover

- Уведомление на email при смене пароля (когда почта включена).

---

## 36. Session fixation

- Regenerate session id после login.

---

## 37. Content Security для inline

- Nonce для необходимых inline скриптов голоса.

---

## 38. Ошибки приложения

- Не показывать stack trace пользователю в prod.

---

## 39. Debug режим

- `RAILS_ENV=production` без `debug` gems exposed.

---

## 40. Third-party subprocessors

- Вести список (хостинг, почта, мониторинг) для политики.

---

## 41. Обучение команды

- Базовый OWASP Top 10 для разработчиков.

---

## 42. Data retention лидов

- Политика архивации/удаления неактивных записей.

---

## 43. Экспорт данных субъекта

- Выгрузка по запросу в машиночитаемом формате.

---

## 44. Шифрование диска

- Провайдер облака full disk encryption.

---

## 45. Key rotation Ollama proxy

- Basic auth пароль смена по расписанию.

---

## 46. Telegram bot token leak

- Немедленно revoke через BotFather, новый токен в secrets.

---

## 47. Логи в ELK

- RBAC на индексы с PII.

---

## 48. Мобильное будущее

- Certificate pinning рассмотреть при нативных клиентах.

---

## 49. AI safety

- Фильтрация персональных данных из промптов внешним сервисам (если когда-либо).

---

## 50. Заключение

Безопасность — сквозной процесс: от кода до операций. Документ уточняется после threat modeling сессии.

---

## 51. Безопасная разработка (SDLC)

Угрозы учитывать на этапе дизайна фичи; чеклист security для PR с касанием auth, файлов, webhooks.

---

## 52. Зависимости фронтенда

При добавлении npm-пакетов (если появится отдельный SPA) — pin версий и audit.

---

## 53. Secrets в логах job

Sidekiq arguments не должны содержать сырой токен; передавать id записи, не секрет.

---

## 54. Clipboard и чувствительные данные

UI не копировать автоматически PII в буфер без действия пользователя.

---

## 55. Print screen и watermark

Не полагаться на технические меры для NDA; договорной уровень.

---

## 56. Двухфакторная аутентификация

Рекомендуется для ролей owner/admin при доступе в продакшен данные.

---

## 57. Блокировка учётки

После N неудачных попыток входа — временная блокировка или captcha.

---

## 58. История паролей

Запрет повторного использования последних N паролей (опционально).

---

## 59. Политика длины пароля

Минимум 12 символов или парольная фраза; не устаревшие требования сложности, противоречащие NIST.

---

## 60. Secure defaults Rails

`force_ssl`, `config.filter_parameters`, отключение подробных ошибок в prod.

---

## 61. Заголовок Permissions-Policy

Ограничить camera/microphone для страниц, где голос не нужен.

---

## 62. Микрофон в браузере

Запрашивать permission только в контексте действия пользователя; не автозапись.

---

## 63. Локальное хранилище

Не хранить токены в localStorage при XSS-риске; предпочтительно httpOnly cookie.

---

## 64. Subdomain takeover

Мониторить DNS CNAME на неиспользуемые сервисы.

---

## 65. SSRF при интеграциях

Валидация URL перед server-side fetch (webhook тесты, импорт по URL — если появится).

---

## 66. Path traversal в скачивании файлов

Использовать signed blob ids, не сырой путь.

---

## 67. Timing attacks на сравнении токенов

Использовать `ActiveSupport::SecurityUtils.secure_compare`.

---

## 68. Insecure deserialization

Не использовать YAML.load с недоверенными данными.

---

## 69. Mass assignment

Strong parameters на всех write endpoint.

---

## 70. Enum injection

Стадии и роли — allow-list, не свободная строка от клиента без маппинга.

---

## 71. Content-Type для API

Отклонять не-JSON там, где ожидается JSON.

---

## 72. HTTP методы

Отключить лишние TRACE/OPTIONS если прокси позволяет.

---

## 73. Security champions

Назначить ответственного за координацию security в команде.

---

## 74. Регулярный review RBAC

Ежеквартально проверять список admin пользователей.

---

## 75. Заключение дополнения

Разделы 51–75 дополняют базовый каркас; при внешнем аудите добавить выводы как новый подраздел или ADR.

---

## 76. Политика хранения логов аудита

Срок хранения `lead_events` и админ-логов не короче требований комплаенса; архив в cold storage с контролем доступа.

---

## 77. Псевдонимизация vs анонимизация

Псевдонимизация — обратимая замена идентификаторов; анонимизация — необратимое удаление связи с субъектом. Выбор — по юридическому заключению.

---

## 78. Обработка запросов субъекта данных

Процедура: идентификация заявителя, срок ответа (например 30 дней), выгрузка/удаление, фиксация в тикете.

---

## 79. Наследие интеграций

При отключении вендора удалить OAuth токены и webhook подписки из БД и секретов.

---

## 80. Безопасность резервных копий на девелоперских машинах

Запрет хранить prod dump на личных ноутбуках; только зашифрованные бакеты с доступом.

---

## 81. Оценка риска поставщика ИИ

Если модель когда-либо внешняя — DPA, регион обработки, запрет обучения на клиентских данных в контракте.

---

## 82. Контроль доступа к аналитике

Дашборды с PII — только для ролей с обоснованием; маскирование по умолчанию.

---

## 83. Физическая безопасность

On-prem Ollama/Whisper — ограничить доступ к серверной комнате; инвентаризация железа.

---

## 84. Увольнение сотрудника

Отзыв SSH, VPN, админок в день выхода; ротация общих секретов при необходимости.

---

## 85. Поставка зависимостей

Проверять целостность gem/npm lock; использовать внутренний mirror при политике air-gap.

---

## 86. Закрытие

Security-позиция продукта эволюционирует; синхронизировать этот файл с `docs/RUNBOOK.md` и `docs/troubleshooter/` после инцидентов.
