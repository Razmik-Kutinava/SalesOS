# PRD — SalesOS / Omni-Agent Revenue OS

**Версия документа:** 0.1 (каркас)  
**Статус:** Draft  
**Языки продукта:** русский, английский (интерфейс, письма, голосовые сценарии)  
**Регионы применения:** Армения, Россия (и смежные B2B-контексты)  

---

## 1. Резюме для руководства

SalesOS — это веб-приложение и Telegram-бот, объединённые общим бэкендом, которые автоматизируют **контур от лида к выручке**: обогащение данных, приоритизацию (скоринг), генерацию и проверку коммуникаций с опорой на **локальный ИИ (Ollama)** и **RAG**, а также **голосовое управление** как основной режим работы «отдела продаж из одного человека». Метрика успеха продукта для владельца — **рост выручки** и снижение потерянных возможностей (лиды «остывают» — система их подсвечивает).

Документ фиксирует **фазирование**: Phase 1 — связка **браузер + голос + Ollama + действия в системе**; Phase 2 — углублённая связка с **уже существующей продакшен-CRM** (БД/импорт/API) и единая балльная логика; Phase 3+ — тяжёлые интеграции, 3D-визуализация лида, расширенная разведка.

---

## 2. Проблема и возможность

### 2.1. Проблема

Один продавец или небольшая команда тонет в рутине: поиск информации о компании, заполнение CRM, черновики писем, напоминания, повторные касания. Часть лидов **теряется** из-за отсутствия единой картины и приоритетов. Внешние LLM-чаты не связаны с CRM и не выполняют **действия** (создать лид, проставить поле, поставить задачу).

### 2.2. Возможность

Локальный Ollama на рабочей машине (в т.ч. ноутбук с **32 ГБ RAM** и современным ARM/x86 CPU) позволяет держать **конфиденциальность** и не платить за каждый токен при интенсивной работе. RAG по материалам вендоров и прайсам повышает качество предложений. Голос снижает трение: пользователь **говорит намерение**, система **предлагает действие** и даёт **красивый approve**.

## 3. Цели и метрики успеха

### 3.1. Главная цель

**Увеличение выручки** за счёт:

- фокуса на лидах с максимальной близостью к деньгам (скоринг);
- снижения «проебанных» лидов (реактивация, напоминания);
- ускорения цикла «сигнал → действие → касание».

### 3.2. Метрики (ориентиры, уточняются при внедрении)

| Метрика | Описание | Примечание |
|---------|----------|------------|
| Выручка / маржа | Главный north star | Вне системы, но в отчётах связывается со сделками |
| Конверсия лида → счёт | Доля лидов с выставленным счётом | Требует дисциплины ввода в CRM |
| Время до первого касания | От создания лида до первого письма/звонка | Phase 2+ |
| Доля лидов с заполненными ключевыми полями | Полнота карточки | После импорта и ИИ-обогащения |
| Доля «остывающих» лидов с повторным касанием | Реактивация | Настраиваемые пороги |
| Ошибки голосового контура | % нераспознанных интентов | Мониторинг качества |

### 3.3. Нефинансовые критерии качества ИИ

- Доля ответов с **опорой на RAG** (где применимо);
- Отсутствие критичных галлюцинаций в **одобренных** пользователем текстах (сэмплинг на ревью).

## 4. Персоны и сценарии

### 4.1. Персона A — Владелец / единственный пользователь (MVP)

Управляет всем контуром продаж, работает в RU/EN, много мобильности. Хочет **минимум кликов**, максимум **голоса** и **быстрых черновиков** с возможностью правки.

### 4.2. Персона B — Администратор (роль в архитектуре)

Настраивает ключи API, промпты, расписания джобов, просматривает логи и здоровье интеграций. В MVP может совпадать с персоной A.

### 4.3. Ключевые сценарии (JTBD)

1. **Создать лид голосом** — назвать компанию, добавить поля голосом; система создаёт запись и запускает цепочку обогащения.
2. **Импортировать сырьё (Excel/CSV)** — загрузка файла; извлечение сущностей; ИИ заполняет поля лида; недостающее — в очередь на парсинг/ручное уточнение.
3. **Голосом дать команду «сделай»** — «найди контакты», «черновик письма EN», «поставь задачу на завтра 15:00».
4. **Диалог в окне** — уточнение стратегии, возражения, варианты письма; затем **approve**.
5. **Скоринг и очередь работы** — список «ближе к деньгам» сверху; отдельно «остывают».
6. **Telegram** — уведомления и резервный голосовой вход (Phase 1 может быть ограниченным).

## 5. Объём продукта и фазы

### 5.1. Phase 1 — Must (каркас «голос + Ollama + действия»)

- Веб-приложение (браузер) как **основной** клиент; стильный UI с минимумом кнопок.
- Захват голоса в браузере → распознавание речи (Whisper / браузерный pipeline — деталь в Architecture) → **Ollama** для интерпретации намерения и генерации текста.
- Выполнение **безопасных действий** в БД: создать/обновить лид, добавить комментарий, задачу, поля (через явные intent-handlers, не «произвольный SQL»).
- Базовый **диалог** и экран **approve** для черновиков.
- Логирование запросов и действий (аудит).
- Минимальная **модель данных** (см. DATA-MODEL.md).
- Ollama локально; конфигурация URL модели в админке.

### 5.2. Phase 2 — Стык с продакшен-CRM

- Подключение к **существующей CRM** (через БД реплики, API или ETL — уточняется): синхронизация лидов/сделок/контактов или миграция нужного подмножества.
- Единая **балльная система** согласованная с CRM (веса: документы, вовлечённость, счёт, оплата, закрывающие — см. раздел скоринга).
- Импорт **Excel/CSV** как промышленный поток (ошибки строк, отчёт).

### 5.3. Phase 3+ — Расширение

- Рассылки (SendGrid/Resend/SES), трекинг открытий, Cal.com.
- Разведка: Serper, Playwright-парсеры (с учётом ToS).
- Snov/Hunter для обогащения email.
- **3D-сцена лида** (абстрактная визуализация: ядро лид — вендоры — продукты — сделки — задачи), вращение мышью.
- «Критик» (второй проход LLM) для фактов и тона.

### 5.4. Явный Out of Scope (на старте)

- Полная замена ERP.
- Гарантированный обход LinkedIn с логином без соблюдения ToS.
- Мультитенантность SaaS для внешних клиентов (архитектурно закладывается осторожно, но не цель MVP).

## 7. Нефункциональные требования

### 7.1. Производительность

- Время ответа Ollama на типовой запрос — в пределах приемлемого UX на целевом ноутбуке (измеряется на этапе POC).
- Веб-страницы — интерактивность без «залипаний» на главных экранах.

### 7.2. Надёжность

- Повторяемые фоновые задачи с идемпотентностью (где есть side-effects).
- Graceful degradation: Ollama недоступен → сообщение пользователю + очередь черновиков.

### 7.3. Безопасность

- Роли `owner`, `admin`, `user` (MVP может свернуть до owner=admin).
- Аудит действий и входов (базовый).
- См. SECURITY-PRIVACY.md.

### 7.4. Локализация

- RU/EN для UI и шаблонов коммуникаций.

### 7.5. Наблюдаемость

- Структурированные логи, correlation id, health endpoints.

## 8. Пользовательские потоки (кратко)

### 8.1. Поток «Создать лид голосом»

1. Пользователь нажимает **единственную очевидную** кнопку записи (или горячую клавишу).
2. Говорит: «Создай лид, компания X, мы продаём им PT».
3. Система показывает транскрипт и **черновик** структуры лида.
4. Пользователь **approve** или голосом уточняет.
5. Запись в БД + событие в аудит + постановка фоновых задач обогащения (Phase 2).

### 8.2. Поток «Импорт Excel»

1. Загрузка файла.
2. Парсинг строк, валидация.
3. ИИ предлагает сопоставление колонок.
4. Пользователь подтверждает.
5. Создание/обновление лидов, отчёт об ошибках.

## 10. Балльная система (черновик v0)

Начальные принципы (уточняются при стыке с CRM):

| Событие / факт | Направление веса |
|----------------|------------------|
| Выставлен счёт | Сильный плюс |
| Оплата / подтверждение оплаты | Сильнее счёта |
| Закрывающие документы | Меньший вес (завершение, не «горячий» сигнал) |
| Наличие ключевых документов в процессе | Плюс |
| Высокая вовлечённость (ответы, встречи) | Плюс |
| Долгое молчание после активности | Минус / флаг «остывает» |

Формула и нормализация — в отдельной спецификации после анализа полей **вашей CRM**.

## 11. 3D-визуализация лида (Phase 3+)

Абстрактная сцена: **ядро** = лид; **орбиты/узлы** = вендоры → продукты → открытые сделки → задачи. Взаимодействие мышью (вращение, фокус на узле). Не MVP.

Технологии кандидаты: Three.js / React Three Fiber / vanilla WebGL — выбор в ADR при старте эпика.

## 12. Риски и допущения

| Риск | Митигация |
|------|-----------|
| Качество локальной LLM | RAG, короткий контекст, критик позже, fallback на облако (опционально) |
| Галлюцинации | Обязательный approve; для фактов — RAG-цитаты |
| ToS сайтов при парсинге | Только разрешённые источники/API; юридическая проверка |
| ПДн | Минимизация данных, сроки хранения, доступы (см. SECURITY-PRIVACY.md) |
| Нагрузка на ноут | Квантованные модели, ограничение concurrency, очереди |
| Сложность слияния с CRM | Поэтапная синхронизация, read-only сначала |

## 14. Критерии готовности Phase 1 (предлагаемые)

1. Пользователь может **создать лид** голосом в браузере и увидеть его в списке/карточке.
2. Пользователь может **добавить комментарий и задачу** голосом с подтверждением.
3. Диалог с Ollama работает с **логированием** и **ограничением** прав (нет произвольных опасных команд).
4. Админка отображает **статус Ollama** и последние ошибки ASR/LLM.
5. Документация запуска (README + RUNBOOK черновик) актуальна.

## 19. Приложение C — Конкуренты и отстройка (кратко)

Классические CRM не дают **голосового операторского** контура с локальным LLM и RAG «из коробки» для соло-продавца. Чат-боты без CRM не ведут к **деньгам**. Отстройка: **единый контур** + **скоринг к выручке** + **локальный ИИ** + **импорт сырья**.
