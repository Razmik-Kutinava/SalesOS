# Интеграции и внешние сервисы

**Версия:** 0.1  
**Статус:** План по фазам (ключи только в env/secret store)

---

## 1. Принципы

- Каждая интеграция за **адаптером** с явным интерфейсом.
- Секреты не в репозитории; ротация документирована в SECURITY-PRIVACY.md.
- Лимиты и ретраи описаны здесь для планирования нагрузки.

## 3. Ollama

**Назначение:** LLM для интерпретации голоса/текста и генерации ответов.

**Подключение:** HTTP к `OLLAMA_HOST` (по умолчанию `http://127.0.0.1:11434`).

**Переменные:**

- `OLLAMA_HOST`
- `OLLAMA_MODEL` (например `llama3.2`, уточнить в ops)

**Лимиты:** зависят от GPU/RAM; нет облачного quota.

**Отказоустойчивость:** деградация — текстовый ввод + сообщение пользователю.

**Документация:** https://github.com/ollama/ollama/blob/main/docs/api.md

---

## 4. Whisper / ASR

**Назначение:** речь → текст.

**Варианты:**

- Локальный `whisper.cpp` / Python binding.
- Облако — только если политика данных разрешает (не MVP по умолчанию).

**Переменные:** `WHISPER_MODEL`, путь к бинарнику при необходимости.

**Лимиты:** CPU/GPU bound; длительность файла < N минут (конфиг).

---

## 5. Telegram Bot API

**Назначение:** второй клиент; голосовые сообщения.

**Переменные:**

- `TELEGRAM_BOT_TOKEN`
- `TELEGRAM_WEBHOOK_SECRET` (path или header)

**Лимиты:** ~30 msg/s per bot (ориентир); файлы до 20MB (голос).

**Документация:** https://core.telegram.org/bots/api

---

## 6. Внешняя CRM (Phase 2)

**Сценарии:**

1. **Read replica** PostgreSQL клиента — только SELECT, VPN/allowlist IP.
2. **REST API** CRM — OAuth или API key.
3. **Файловый экспорт** — реже, batch импорт в SalesOS.

**Переменные (пример):** `CRM_API_BASE_URL`, `CRM_API_KEY`, `CRM_DB_URL` (read-only).

**Лимиты:** по договорённости с CRM; backoff при 429.

**Стык данных:** таблица `crm_id_mappings` (см. DATA-MODEL.md).

---

## 7. SMTP / транзакционная почта

**Назначение:** письма из системы (напоминания, follow-up).

**Провайдеры:** Resend, SendGrid, Mailgun, корпоративный SMTP.

**Переменные:** `SMTP_URL` или по отдельности host/user/pass.

**Phase:** минимум в Phase 2 для продакшен-качества.

**Лимиты:** дневные квоты провайдера; bounce handling позже.

---

## 8. Cal.com (Phase 3)

**Назначение:** запись на звонок из карточки лида.

**Интеграция:** OAuth + webhooks на отмену/создание.

**Ключи:** `CALCOM_API_KEY`, webhook signing secret.

---

## 9. Serper / Google Search API (Phase 3)

**Назначение:** обогащение компании по запросу.

**Переменные:** `SERPER_API_KEY`.

**Лимиты:** платный tier; кэшировать результаты.

---

## 10. Парсинг сайтов (Phase 3)

**Паттерн:** headless или HTTP fetch + readability; уважать robots.txt и правовые ограничения.

**Лимиты:** concurrency cap в job queue.

---

## 11. Stripe (опционально)

**Назначение:** оплата подписки SaaS, если модель изменится.

**Переменные:** `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`.

---

## 12. Active Storage / S3

**Назначение:** документы лидов, импорты.

**Переменные:** `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `S3_BUCKET`, или совместимый MinIO.

**Phase 1:** локальный disk допустим; прод — object storage.

---

## 13. Redis (если Sidekiq)

**Переменные:** `REDIS_URL`.

**Назначение:** брокер очередей и кэш.

---

## 14. Solid Queue / Solid Cache

Встроенные в Rails 8; БД как бэкенд — без Redis возможен trade-off.

---

## 15. Мониторинг (опционально)

Sentry: `SENTRY_DSN`  
Application Insights / другой APM — по политике заказчика.

---

## 16. Карты / геокодинг

Не в MVP; при появлении — отдельный ключ и политика хранения.

## 17. Push / FCM

Не в MVP.

## 50. Active Record Encryption (опционально)

При требовании регулятора к полям PII — встроенное шифрование Rails на уровне колонок; ключи в KMS.

---

## 51. Интеграция с корпоративным LDAP/SSO

Не Phase 1; при запросе — SAML/OIDC провайдер, отдельный ADR и сроки.

---

## 52. Webhook входящий от CRM

Если CRM пушит изменения: endpoint с HMAC, idempotency по `event_id` от CRM.

---

## 53. Лимиты Ollama при shared host

Очередь запросов на уровне приложения (семафор), чтобы не убить GPU соседними джобами.

---

## 54. Мониторинг квот облачного ASR

Если когда-либо включён облачный ASR — дашборд usage и алерты 80% квоты.

---

## 55. Интеграция с 1С / Bitrix

Через REST модули или коннектор партнёра; маппинг полей в `crm_id_mappings` и staging-таблицах.

---

## 56. Google Workspace / Microsoft 365

Календарь и контакты — только после явного scope OAuth; не смешивать с лидом без согласия.

---

## 57. Slack уведомления

Incoming webhook URL в secrets; сообщения без PII или в приватный канал.

---

## 58. Datadog / New Relic

API keys в secrets; отключить авто-инструментацию чувствительных параметров запроса.

---

## 59. GitHub (CI только)

`GITHUB_TOKEN` для Actions; не использовать для продакшен runtime.

---

## 60. Docker Hub / GHCR

Pull credentials для Kamal; rotate при утечке.

---

## 61. Let's Encrypt

Автообновление сертификатов на edge; мониторить срок действия.

---

## 62. MinIO для dev

S3-совместимый локально; те же ENV имена что и для AWS для переносимости.

---

## 63. Разделение staging/prod ключей

Запрет на использование продакшен Telegram токена на staging.

---

## 64. Логирование вызовов CRM

Сохранять `duration_ms`, `status`, `correlation_id`; тело запроса/ответа — только в debug и без PII.

---

## 65. Конечные точки Ollama для продакшена

Только `/api/generate` или `/api/chat` из allow-list; остальное закрыть на reverse proxy.

---

## 66. Защита Whisper от path traversal

Входные файлы только в tmp с random именем; удалять после обработки.

---

## 67. Интеграция с почтовым провайдером webhooks

Bounce/complaint webhooks для подавления рассылок на плохие адреса.

---

## 68. Сезонность нагрузки Telegram

Маркетинговые кампании могут дать всплеск — масштабировать workers заранее.

---

## 69. Geo-распределение

Если CRM в одном регионе, приложение ближе к CRM снижает latency sync.

---

## 70. Выбор региона облака

Учитывать data residency и latency к Ollama (часто on-prem).

---

## 71. Сторонние виджеты на сайте

Не встраивать в authenticated app без CSP и review.

---

## 72. Экспорт в BI

Read-only replica + view для отчётов; не нагружать primary.

---

## 73. NTP и время

Синхронизация времени на всех хостах для корректных webhook подписей и аудита.

---
